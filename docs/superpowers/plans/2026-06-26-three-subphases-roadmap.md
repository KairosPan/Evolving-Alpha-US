## 1. Recommended BUILD ORDER

**Build order: §6 Memory → §5 Self-Evolution → §4 Conversational Workspace** — with a tiny **Foundation step (F0)** extracted ahead of all three (see §3).

```
F0  Shared Foundation (provenance schema + SQLite convention) — ~½ day, blocks §5 & §6
 │
 ├─1st──  §6 MEMORY (episodes)         ← touches apply_credit ONLY
 │
 ├─2nd──  §5 SELF-EVOLUTION (provenance/conflict) ← touches try_apply_op + its 4 callers
 │
 └─3rd──  §4 CONVERSATIONAL WORKSPACE  ← consumes BOTH (version-ref + the converse 3-tuple caller)
```

### Why this order

**§6 Memory goes 1st (not 3rd despite "wanting" SQLite).**
- Its only shared write is `apply_credit` (`credit.py`). §5 does **not** touch `apply_credit` — it touches `try_apply_op` (same *package*, different *function*). So §6 and §5 are **non-colliding within `alpha/refine/`**; the `touches_shared` "alpha/refine/ overlap" worry is between §6 and the **deferred §6.6 auto-adjust plan**, not §5. They can even be parallelized, but §6 first lets F0's SQLite convention get exercised by a real consumer before §4 reuses it.
- §6 is the **most self-contained** (all new `alpha/memory/` modules, one optional kwarg on `apply_credit`, one optional `LoopConfig` field). Lowest blast radius → de-risks the SQLite substrate decision early.

**§5 Self-Evolution goes 2nd.**
- It changes `try_apply_op`'s **arity** (2-tuple → 3-tuple) and rewrites **4 caller sites** (`refiner.py`, `meta/agent.py` ×2, `converse/tools.py`). This is the single largest cross-cutting edit. Doing it before §4 means §4's NEW `converse_project` is written against the already-final 3-tuple gate — it never inherits a stale 2-tuple assumption.
- It **produces** the `EditProvenance` schema and `parent_checkpoint_version` field that §4's H-version provenance story must align with. §4 must not invent a *second* version-ref vocabulary.

**§4 Conversational Workspace goes 3rd (last).**
- It is a pure **consumer**: reads `SnapshotStore.latest()`/`.load(pin)` (untouched), reuses `_atomic_write`/`_path`, and its one shared edge into `try_apply_op` is via `converse/tools.py::make_gated_write_tool` — which §5 already converted to the 3-tuple. If §4 went earlier, §5 would have to re-edit `converse/tools.py` again.
- It depends on the **provenance story being settled** (F0/§5): a "pinned" project's gated writes must produce `EditProvenance(path='teaching', proposer='hermes')` — that proposer enum value is owned by §5.

## 2. Concrete SHARED CONTRACTS (build once, all three adopt)

### Contract A — Version-ref & Provenance (resolves cross-cut #1)

**ONE version authority: `SnapshotStore` integer versions.** There is no second version concept. Both the §5 `EditRecord.provenance` block and the §4 per-turn "H-version provenance" reference the **same `int`** returned by `SnapshotStore.latest()` / accepted by `.load(version)`.

Two **distinct, non-overlapping** roles for that int — name them explicitly so the synthesizer never conflates them:

| Field | Owner plan | Meaning | Where stamped |
|---|---|---|---|
| `EditProvenance.parent_checkpoint_version: int \| None` | **§5** | the H-version the gate was operating *on* when an edit was applied (audit of *what produced an edit*) | inside `try_apply_op`, read from `HarnessManager.latest_version()` if a manager is reachable, else `None` |
| `ProjectTurn.h_version: int \| None` | **§4** | the H-version a *conversation turn ran against* (audit of *what a turn saw*) | in `converse_project`, `= project.h_pin if pinned else snapshots.latest()` |

**Both are reads of the same `SnapshotStore`. Neither plan modifies `SnapshotStore` or `HarnessManager`.** §5 *defines* the schema; §4 *reuses the same int authority* for a different field. This is the "one consistent story": **SnapshotStore owns versions; §5 owns the provenance *block*; §4 owns the per-turn *ref*; both point back to SnapshotStore ints.**

**Canonical `EditProvenance` schema (built in F0, lives in `alpha/harness/edit_log.py`):**
```python
class EditProvenance(BaseModel):
    model_config = ConfigDict(frozen=True)
    path: Literal["self_study", "teaching"]
    proposer: Literal["refiner", "forge", "sonia", "hermes"]
    evidence_ref: dict | None = None
    reflection_lm_id: str | None = None
    reflection_seed: int | None = None
    human_approver: str | None = None
    parent_checkpoint_version: int | None = None   # ← the SnapshotStore int (shared authority)
    resolution: str | None = None
```
- `EditRecord` gains `provenance: EditProvenance | None = None` (additive, back-compat).
- The B-WIDE converse write face tags `path="teaching", proposer="hermes"` (confirm in open-decisions).

### Contract B — SQLite substrate (resolves cross-cut #2)

**Decision: KEEP THEM SEPARATE. JSON for conversation/project persistence; SQLite (`brain.db`) ONLY for episodes.** Grounded justification:

- Every existing store in this tree is **atomic-JSON-by-id** (`SessionStore`, `LiveBrainStore`, `DecisionStore`, `VerdictStore`, `SnapshotStore`). §4's `ProjectStore` should mirror that — it carries a `ChatMessage` list + turn log, a document workload, not a query workload. There is **no FTS/recall requirement** on conversations.
- §6's `EpisodeStore` genuinely needs **FTS5 + `learned_asof <= :asof` range queries + BM25 recall** — a query workload SQLite is built for. JSON cannot serve `search()`/`recent()`.
- The spec's §4.1/§8 reserves Hermes's `state.db` for the **vendored Hermes core that does not exist yet** (Phase-0 spike gates it). Introducing SQLite into §4 now would pre-commit to a schema Hermes will own. **§4 ships JSON with a documented seam** ("swap message log to Hermes `state.db` post-spike").

**Single shared convention both adopt (so "one SQLite module convention" is satisfied even while separate):**
- **File path env**: `ALPHA_BRAIN_DB` → `brain.db`. §6 owns the only SQLite file today. `EpisodeStore.open(path, create_if_missing=True)` + `.in_memory()` for tests.
- **JSON root env**: `ALPHA_PROJECTS_DIR` (§4), mirroring the existing `ALPHA_WEB_DECISIONS_DIR` / `ALPHA_CONFLICTS_DIR` naming family.
- **When Hermes lands**, `brain.db` is the natural host for *both* the episodic store *and* the vendored `state.db` session tables — but that consolidation is a **post-spike migration**, explicitly out of scope. The seam (env-driven path, `open()` classmethod) makes it cheap.

### Contract C — Table / store conventions

- **JSON stores** (§4 `ProjectStore`, §5 `ConflictQueue`): mirror `SessionStore` **verbatim** — `_atomic_write` (re-implemented locally to avoid `alpha.meta` coupling, per the plans), `_path` with `.resolve()` + `is_relative_to` traversal guard (ids arrive from URL params), `model_validate_json`, newest-first `list()` tolerant of garbage files.
- **`brain.db` schema** (§6 only): `episodes(episode_id PK, …, learned_asof TEXT NOT NULL, superseded INTEGER DEFAULT 0)` + INDEX on `learned_asof` + FTS5 `episodes_fts(reflection_text, narrative)`. The **`learned_asof <= :asof` PIT mask lives in the store query methods**, mirroring the already-shipped lesson PIT mask at `retrieval.py:25-52`.

# Appendix — the 3 plans (task lists)


## Conversational workspace — project = {resumable conversation + git-backed workspace + H-version provenance}

Builds the §4 PROJECT/WORKSPACE layer for the Hermes re-base: a project binds (a) a resumable conversation, (b) a per-project git-backed workspace dir for artifacts, and (c) a per-turn H-version provenance ref into SnapshotStore. The existing `alpha/converse/` package (agent/loop/registry/tools — Phase-1A/1B conversational face) is stateless today: `converse()` builds a fresh registry, runs `run_conversation`, and discards everything. This plan adds a thin persistence + workspace + provenance layer ON TOP, mirroring the proven JSON-store idioms (`DecisionStore`, `SessionStore`, `_atomic_write`, the `SessionStore._path` traversal guard) without touching the day-agent (`LLMAgentPolicy.decide`) or `InnerLoop`.

STORAGE RECOMMENDATION — JSON, not SQLite. The spec's §4.1/§8 vendor table reserves SQLite+FTS5 for the *vendored Hermes* `state.db` (session/message persistence) and for the future episodic `brain.db` (§6, a different plan's territory). But the vendored Hermes core does not exist yet (Phase-0 spike gates it), and every existing store in this repo is atomic-JSON-by-id (`DecisionStore`, `SessionStore`, `VerdictStore`, `SnapshotStore`, `LiveBrainStore`). To stay ADDITIVE, ship the project/conversation store now as a `ProjectStore` mirroring `SessionStore` exactly (atomic write, `_path` traversal guard, `model_validate_json`), with a documented seam to swap the message log to Hermes's `state.db` once the spike lands. This avoids introducing a SQLite dependency the 584-test suite doesn't have, and avoids pre-committing to a schema Hermes will own. Each conversation persists as one `Project` pydantic doc (a `ChatMessage` list + tool-call log + per-turn provenance refs).

GIT WORKSPACE — a `Workspace` helper wraps `git init` once per project under `<projects_root>/<project_id>/workspace/`, and `commit_artifact(path, data, message)` writes the file then `git add`+`git commit` (subprocess, no new dep — git 2.52 confirmed available; mirror the `scripts/` subprocess idiom). `DecisionPackage` artifacts (§4.2) land here as `<date>.json` exactly like `DecisionStore.put`, then are committed, giving them git provenance. Determinism in tests via isolated git author/HOME env + a fixed date, all under tmp_path.

PROVENANCE — every turn stamps `h_version = SnapshotStore.latest()` (or a project pin) at the moment the turn runs, stored on the persisted turn. A project may optionally pin an H-version; `converse_project()` loads `HarnessState` from `SnapshotStore.load(pin)` when pinned, else uses the live harness and stamps `latest()`. This reuses the existing version authority (`SnapshotStore`/`HarnessManager`) unchanged.

All work is new files under `alpha/converse/` (project.py, store.py, workspace.py, session.py) + new tests under `tests/converse/`. The day-agent, InnerLoop, try_apply_op, SnapshotStore, and the existing `converse()` entrypoint are untouched (a NEW `converse_project()` wraps `run_conversation`; the old `converse()` stays for back-compat and its 4 tests stay green).
