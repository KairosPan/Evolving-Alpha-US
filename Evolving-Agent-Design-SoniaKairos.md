# SoniaKairos — Evolving Agent Design Principles

This document designs **Kairos** — the agent that gets modified; every mechanism below exists to make modifying an agent safe, auditable, and reversible. **Sonia**, the teacher agent who (together with the user) does the modifying, has her own kernel and her own Body, and is designed after Kairos — this document pins only her boundary properties. See *Sonia: The Teacher Agent* under the Evolution Layer.

## First Founding Principle: Kairos Acts, Sonia Evolves

Kairos moves along two axes; this design treats them oppositely and splits them cleanly **by principal**:

- **World-capability** — what Kairos can *do* to the world: edit external repositories, run arbitrary code, call connectors, open tunnels. **This axis is maximized.** A worker with a small action space is a weak worker; nothing in this document exists to shrink it.
- **Self-authority** — what Kairos's own definition (the Body) may *become*: **Kairos holds none of it.** Kairos does not edit its definition, not even ephemerally within a session; every change to the Body is **Sonia's to propose, the user's to approve, the Applier's to land** (see *Evolution Layer*). A Kairos session produces work and evidence (activity traces) — never a change to Kairos itself.

The invariant every mechanism below assumes and none may violate:

> **Every change to Kairos's definition lands through the Applier, and only two hands may send it there (amended 2026-07-08, re-review): a Sonia proposal that passes deliberation → user approval → Applier, or the User's own direct edit — stamped and audited, no deliberation in between. A Kairos session produces evidence and work, never a persistent change to itself. A session's blast radius dies with the session.**

The split is what makes the large action space *affordable*: because a session cannot alter Kairos at all, a compromised or mistaken activity session poisons at most its own run, and the next session starts from the clean, approved tip (or, when the tip itself is being killed on sight, from the bare kernel floor — safe mode, see *Boot & recovery* — never from an unapproved state).

If Kairos could edit itself — even ephemerally — the same large action space would become a self-poisoning channel the moment an injected adaptation were replayed. Separating the actor (Kairos) from the evolver (Sonia) is the safety belt that lets the actor's action space be large.

(One corollary the *Security Boundary* completes: the blast-radius invariant holds for Kairos's *definition* by this split, but for *data confidentiality* only once sandbox egress is controlled — an injected session that can exfiltrate private memory causes harm that outlives it.)

> **Deferred (2026-07-06):** Kairos **session-local self-adaptation** — editing its own prompt / skills / tools mid-session to unblock itself — is shelved, not chosen-against-forever. Accepted cost: Kairos cannot self-unblock mid-task and must fail into Sonia's offline refinement; the paper's ~500-turn-stall evidence is the accepted risk of routing all evolution offline, to be re-weighed only with evidence from Kairos's own traces (not the single predecessor anecdote). Revisit trigger: offline-only refinement shown too slow on Kairos's real workload. (Amended 2026-07-07: the evidence-curation gate adds curation latency to this same offline path — that cost is recorded here — and the curation-starvation condition, *Dreaming*, merges into this same revisit trigger.)

This design rests on **two founding principles**, and the front matter states both. The first is the governance axiom above — *who* may persistently change Kairos (Sonia proposes and the user approves, or the User edits directly — amended 2026-07-08; the Applier lands either; Kairos never). The second is the structural axiom that makes the first *enforceable*: **Decouple** (next) — Kairos is seven independent components, each able to fail or be replaced on its own. The two relate as end and means (reworded 2026-07-07, superseding the earlier "neither is prior" phrasing, which contradicted the very next clause): the first states the governance goal, the second is its enforcement precondition — the actor/evolver split is enforceable *only because* the components are physically separate: a powerful, disposable sandbox Kairos acts through; a deliberation-gated Body only the Applier writes; credentials sealed in the Vault, never in the sandbox; the raw record in the Session beneath both. Together the two are the precondition the rest of this document elaborates. (*Harness = Kernel + Body*, later, refines the fixed/evolvable line **between** two of these seven components — the Kernel and the Body, which together are the harness — and is subordinate to these two founding principles, not co-equal with them.)

## Second Founding Principle — Decouple: Kairos = Model + Kernel + Body + Memory Stores + Session + Sandbox + Vault

Seven independent components; each can fail or be replaced on its own (the enforcement precondition of the First Founding Principle above — end and means, reworded 2026-07-07; the actor/evolver split is enforceable only because these components are physically separate):

- **Model** — the weights. Swappable; all training decisions are v4-gated.
- **Kernel** — the harness's fixed part and the system's fixed function: the loop that assembles the context window, calls the model, and routes tool calls, plus the kernel services around it (watchdog, Applier, vault proxy, trace logger). Fully stateless and in-system immutable: if it crashes, a new one restarts with `wake(sessionId)` and continues from the event log; if it must change, the operator ships a release. See *Harness = Kernel + Body* below.
- **Body** — the harness's evolvable part: the six-component surface (system prompt's evolvable segment, sub-agents, workflows, connectors' evolvable fields, skills/tools/plugins, memory) that evolution rewrites. It needs its own independent home — the **Body Store**, one git repository per Kairos instance: the kernel is stateless, sandboxes are disposable, and the session stores only events, so without it one sandbox TTL expiry is a silent factory reset of everything evolution built. Write access to the Body remote is the **Applier's alone**; sandboxes get a read-only checkout and no push credential (see *Security Boundary* and *Body Persistence & Versioning*). See *Body Persistence & Versioning* under the Evolution Layer.
- **Memory Stores** — the memory component's storage home. Since the 2026-07-06 pivot, Kairos does not write memory; **Sonia writes memory content as deliberation proposals**, so *both content and structure* change via deliberation → user approval → Applier — or (amended 2026-07-08) via the User's direct edit through the same Applier — one write regime with the rest of the Body, two triggers. The store of record is **Mem0** (decided 2026-07-09; a separate vector store, not in-repo files — see *Memory Design → Substrate*), reconstructable from the git-committed memory journal (see *Cross-store consistency → Memory leg*); its separation is substrate-driven, no longer any ungated hot-path write (that path is retired). Content is **typed**: facts/state live here; procedures/behavior live only in Sonia-authored executable components (skills/workflows). (Component status clarified 2026-07-07: what earns this a place in the founding enumeration is its content type and its write governance — Sonia proposes memory content via deliberation, or the User edits it directly (2026-07-08) — not the substrate; the store is Mem0, reconciled from the journal per *Cross-store consistency*, without changing this list.) Failure semantics (added 2026-07-07): a store that cannot be mounted or reconciled to the tip's declaration quarantines the session rather than running degraded — see *Cross-store consistency → Boot reconciles to tip*.
- **Session** — an append-only event log that exists outside every other component. The loop writes to it through `emitEvent()` and reads from it through `getEvents()`. No matter how the loop or sandbox fails, the record remains. Events are principal-origin-stamped and hash-chained at intake (decided 2026-07-06; see *Trust Roots & Principal Authentication*). The raw record beneath the system: neither model nor harness.
- **Sandbox** — an isolated, disposable execution environment (hands) for running code and editing files. The loop calls it like any other tool: `execute(name, input) -> string`; if one dies, start another, and the loop passes the failure back to the model as a tool error. It mounts the Body repo **read-only** and holds no push credential to the Body remote (only the Applier writes it); its in-session working copy is ephemeral and never persists. Its outbound network is **default-deny**, routed through the kernel egress proxy (see *Security Boundary → Sandbox egress*). Memory stores are separate mounted stores under deliberation governance, not written from here.
- **Vault** — the credential store + token-injection proxy. Secrets never enter the loop or the sandbox; OAuth grants are versioned vault-side, hard-deletable, and cross-referenced from the Body repo's lockfile. Failure semantics (added 2026-07-07): if the vault or its proxy is unreachable at runtime, credentialed calls fail as ordinary tool errors — fail-closed, no cached or degraded credential path; the session continues with that capability absent.

Where the old list's names went: **Agents** are not a component but the composition — `Agent_t = Kernel(Model, Body_t, Session_t)`, acting through sandboxes with memory stores mounted; **Environments** = Sandbox; **Skills** are a Body component, not a top-level store.



## Harness = Kernel + Body

**Harness** names everything around the model: every piece of code, configuration, execution logic, and content loaded to configure the agent (memory included — definition completed 2026-07-07) that isn't the model itself — fixed and evolvable alike. Agent = Model + Harness. Inside the harness runs the one line this whole document exists to govern:

> **Harness = Kernel ∪ Body. The Body is the harness's evolvable part; the kernel is the part evolution can never touch; the session log is the raw record beneath both. Evolution rewrites Body_t — never the kernel.**

Two placement tests draw the kernel/Body line inside the harness. For executable machinery they coincide; kernel-hosted governed data rows, kernel-written observability/ledger rows, and user-owned records are the enumerated exception classes, where agent-unwritability — not multiplicity — decides placement:

1. **Binding** — always present, shipped hard-wired → kernel; loaded to configure → Body.
2. **Multiplicity** — identical across all Kairos instances → kernel; differentiates one instance/tenant → Body.

**The strata — the harness as structure, the kernel as function:**

```
Harness_t = Kernel ∪ Body_t                    — the whole wrapper; its evolvable part is the Body
Agent_t   = Kernel(Model, Body_t, Session_t)   — the kernel is the fixed function; the Body is its evolvable argument
```

- **Kernel** — the harness's fixed part, and the fixed *function* of the system: the loop with its context-assembly machinery, watchdog/kill-switch, Applier, vault + token-injection proxy, trace logger, orchestration quotas, static policy gates, and the always-present kernel prompt stub. Coextensive with the Immutable Kernel's mechanism stratum: **kernel ≡ the operator-release-only stratum**, identical across all Kairos instances. (Sonia runs on her own kernel — see *Sonia: The Teacher Agent*.) (Kernel-hosted governed data rows — egress registry entries, scope-whitelist entries, lockfile pins, connector kernel fields, the user's charter and correction records, the never-codifiable decision-class list — are not mechanisms: agent-unwritable cargo, changed only through their designated gates.)
- **Body_t** — the harness's evolvable part: the six-component surface (system prompt's evolvable segment, sub-agents, workflows, connectors' evolvable fields, skills/tools/plugins, memory — content and structure alike). This is what the continual-harness paper calls "the harness"; in this document that word always means kernel + Body together, and the paper's usage maps to Body. Per-instance, versioned, evolvable — and still harness.
- **Session_t** — the append-only event log: the raw record beneath the system, neither model nor harness. Nothing edits it; every learning mechanism reads from it.
- **Model** — the weights (v4-gated).

Evolution rewrites the kernel's **argument**, never the kernel. "Self-improving harness" is literally true — the harness contains an evolving part — and precisely bounded: the evolving part is exactly Body_t. Structure and components enter via the teaching pipeline (deliberation → user approval → Applier; see *Sonia: The Teacher Agent*) or via the User's direct edit through the same Applier (amended 2026-07-08); memory content enters the same two ways — Sonia proposes it or the User edits it, the Applier lands it (the old ungated write-waist is retired; see *Memory writes — one path*). Vocabulary stays isomorphic to the permission model — one name per write regime, with the exceptions enumerated rather than implicit (kernel-hosted governed data rows; kernel-written observability/ledger rows; user-direct records — which since 2026-07-08 include direct Body edits, landed by the Applier):

| Stratum | Write regime | Learning at this layer |
| --- | --- | --- |
| Model | training pipeline only (v4, gated) | SFT/RL; catastrophic forgetting is the noted risk — collect traces, build no pipeline yet |
| **Kernel (the fixed function)** | operator-only, human release | exists — traces expose kernel deficiencies — but routes **out-of-system** as operator-channel requirements; an unreviewed in-system self-improvement loop over kernel code may not run here |
| **Body** | structure, components & memory content: teaching pipeline (Sonia proposes → user approval → Applier) plus the User's direct edit through the same Applier (amended 2026-07-08) — one writer, two triggers; session-local self-adaptation: **deferred** | the entire Evolution Layer: scheduled dreaming offline → proposals; scope labels set the granularity |
| Session | kernel appends (loop + trace logger); nothing edits | not learned — what every other stratum learns *from* |

(User-owned records — the preference charter, correction records — are kernel-side data under user-direct write: one more write regime, in-system immutable to both agents.)

**Fixed points this definition pins down:**

1. **Terminology.** The continual-harness paper's "harness" = our Body — the evolvable part only; its fixed "minimal harness" ≈ our kernel with an empty Body. The version stamp is **body-version** ≡ the Body commit hash (simplified 2026-07-07: the lockfile is a committed file, so the hash already pins it — the earlier "commit hash × lockfile" tuple was informationally redundant; the lockfile's role is bridging the committed declaration to vault-side grant versions, and *effective* state can still degrade from the declared version, which boot reconcile and the restorability pre-flight detect); traces carry the full attribution tuple **(body-version × model-id × kernel-version)** (decided 2026-07-06, resolves review §2.5) — a model swap or kernel release moves behavior at least as much as a Body edit, and incident attribution that cannot see them will indict innocent commits.
2. **Kernel prompt stub.** If 100% of the system prompt were Body, one approved edit could strip the security invariants or the deliberation-protocol instructions that govern approval itself. The system prompt therefore splits the way connectors do: kernel stub (identity, security invariants, deliberation protocol — operator-owned, always present) + evolvable Body segment. **Mount notes are kernel-rendered, not kernel-authored (decided 2026-07-06, resolves review §1.4):** the two placement tests disagree on them — Binding says kernel (always present), Multiplicity says Body (mount layout differentiates instances) — so they split the way the context window does (fixed point 5): the note *template and its delivery* are kernel mechanism; the note *content* is a per-call projection the kernel renders from the Body repo's memory config. An approved store change can therefore never leave the stub's notes stale, and nothing ever needs to write kernel text to refresh them.
3. **Kernel-layer learning routes to the Operator.** Instruction-extraction target classification is three-way: Kairos-Body / Sonia / kernel — the latter two go to the operator channel as requirements, never as deliberations.
4. **The kernel is the capability floor, not merely evolution's guardrail.** Filesystem, code execution, sandbox + verification tools, memory plumbing (the mechanism only — the memory component itself, content and structure, is Body), context management, long-horizon machinery all live in the kernel. The paper's Flash-Lite sign inversion (minimal harness 20% vs every evolved variant 3–13%) is the evidence: the floor must never depend on evolution.
5. **The context window belongs to no stratum.** It is an ephemeral per-call projection the kernel computes from Body + Session — companion principle to "Session Is Not the Context Window": the window's content comes from Body and Session, its assembly is kernel code. "Context" is not a stratum name in this design; what is commonly called context or memory is the Body, configuring the kernel from outside it.
6. **The Sandbox splits across the kernel/Body line.** Its control plumbing (the `execute()` contract, provisioning, snapshot/fork machinery) is kernel mechanism — harness; a running container instance is a disposable runtime resource, no more harness than a process is a program; any evolvable sandbox image/config field is Body.

## Security Boundary: Two-Class Credentials — the Work Token Is Contained; Everything Else Never Enters

(Retitled 2026-07-07 — the old title "Credentials Never Touch the Sandbox" overclaimed against its own first bullet; evidence: `docs/research/2026-07-07-credential-isolation-codex-deepagents-hermes.md`.)

- **Git access — work repos**: For repositories Kairos edits *as a task* (a user's project, a PR), the CMA pattern holds: a token is cloned into the sandbox and written into the git remote config; the sandbox can push and pull. **"The agent never sees the token" is retired as a claim (2026-07-07, credential-isolation study):** the token is sandbox-resident and readable by any code the agent executes there (`cat .git/config`) — no runtime-hiding mechanism exists here or in any surveyed system (DeepAgents' identical pattern is flagged exfiltratable by its own docs; Hermes treats this URL shape as a secret to redact) — and it outlives the session unless rotated. Containment, not secrecy, is the load-bearing claim — **two boundaries, promoted to the main promise:** the token is repo-scoped to the work repo only, and it must be *physically incapable* of reaching the Body remote (separate host / separately-scoped token / Body remote unreachable with the work token — decided 2026-07-06; otherwise the two-class split is only on paper); with sandbox egress allowlist-only, a stolen token also has only one useful destination. Work repos are outside the trust boundary of Kairos's own definition. Per-session token minting is deferred hardening (see *Work-credential hardening*); rotation/expiry stays deferred with Tier 2 (see *Body write-path hardening*).
- **Git access — the Body repo (decided 2026-07-06)**: The Body repo is **not** a work repo. It is mounted **read-only**; the sandbox gets **no push credential** for it. Persisting to the Body tip is exclusively the Applier's act — enforced by ordinary access control (the Applier process holds the only write path; the sandbox holds none), which is what closes the critical write-path bug. So an injected activity session has nothing to push with, and `always tip` can only ever mount Applier-landed commits. Cryptographic enforcement (a pre-receive hook verifying signatures) is deferred — see *Body write-path hardening*.
- **External tools**: Tools connect through the MCP protocol, with OAuth tokens stored in a secure vault. The model calls MCP tools through a dedicated proxy, which uses the session token to fetch credentials from the vault before making the request. Neither the loop nor the sandbox knows that the credentials exist — for this class, "never touches" holds literally. (A stray vendor-voice duplicate of this bullet was removed 2026-07-07.)

### Boundary Extensions for an Evolving Body (added 2026-07-04)

The bullets above defend **token secrecy** only. Once the Body is evolvable, three surfaces open that secrecy does not cover (from the 2026-07-04 blindspot analysis of the continual-harness paper, `references/continual-harness.txt`):

- **Destination integrity (confused deputy)**: nothing above prevents a token from being *attached to the wrong place*. A one-line approved change to a connector's base URL / token audience / OAuth scope silently re-points the user's OAuth consent at a new counterparty — the credential never needs to be seen. Therefore connector definitions split into evolvable fields (server choice, tool config, descriptions, timeouts, retries) vs kernel fields (base URL, token audience, OAuth scope — user-personally-approved only), and the egress set is a **gated registry, not a static list**. Adding a connector = adding an egress path = a deliberation requiring user approval, never a self-service act.
- **Poisoned executable components persist across sessions**: memory stores got structural prompt-injection defenses (read_only mounts, immutable versions); skills, sub-agents, workflows, and plugins — which *execute* — must get no less: versioned, session-attributed, gated behind deliberation. A skill written by one poisoned session is otherwise a cross-session backdoor.
- **Training-data egress**: teaching dialogue, corrections, and memory content flowing into third-party training pipelines is an egress path this section never enumerated. Erasure does not propagate into already-cut shards/adapters. Default: user-instruction material and deliberation records are **not-trainable** unless explicitly released (details gated on v4).
- **Body-repo write integrity (added 2026-07-06)**: token secrecy prevents the credential from being *seen*; it does nothing to prevent a write from being *forged*. The prior surfaces assumed the Applier is the sole writer but did not enforce it — the sandbox held push access. Tier-0 fix: **access control, not cryptography** — the sandbox has no write path to the Body remote (read-only mount, no push credential), and the Applier (a small kernel service, off the prompt-injection surface) is the only writer, so there is no path to forge a Body on. Cryptographic provenance (signed commits, a pre-receive hook, user-approval signatures) is a later layer that pays off only against a compromised Applier / leaked push path / distrusting multi-party setting — **deferred**, see *Body write-path hardening*.

### Sandbox egress: default-deny + destination allowlist (added 2026-07-06)

The bullets above defend token secrecy and destination integrity for *credentialed* egress. But the sandbox runs arbitrary code and can `curl` any host or open a tunnel with no connector and no credential — the two highest-bandwidth exfiltration channels, ungoverned. Because Kairos's action space is maximized (edit external repos, run code), this hole is at its largest, and it is where an injected session causes external harm that outlives it — so egress control *completes* the First Founding Principle's blast-radius invariant, it is not an add-on. Design calibrated against Codex Cloud (borrow), Hermes-agent (avoid), DeepAgents (don't delegate). Core lesson: **enforce below the tool layer** — since the sandbox runs arbitrary code, any tool-wrapper network guard is bypassable by construction (Hermes's `url_safety.py` covers only native web tools; `curl` walks past). Enforcement sits at the sandbox network namespace, where arbitrary code cannot route around it.

- **Default posture — drop-all outbound.** The sandbox runs in a network namespace with an nftables default-drop policy; no process reaches any external host except through the kernel-side **egress proxy** (the sole route out).
- **Proxy enforcement** (from Codex Cloud): a hostname allowlist (raw-IP / non-allowlisted hosts dropped); private / link-local / metadata-IP blocking *even for allowlisted hosts*; DNS failure/timeout = block (fail-closed); optional per-host read-only HTTP-method restriction — with the recorded caveat that it does **not** stop GET-string exfil.
- **One governance surface** — the allowlist is *derived from* the connector registry, not parallel to it: approving a credentialed connector (already a deliberation) auto-adds its hostname(s); un-credentialed dependency hosts (PyPI, npm, GitHub, container registries) are a named preset approved once as a registry-style deliberation; a custom host = the same deliberation. So `curl` and tunnels are gated by the *same* mechanism that already gates connectors — no second approval UX.
- **Port-tunneling under the same gate**: listening on a port (inbound) stays a free sandbox feature; the tunnel's *outbound leg* is subject to the allowlist — a tunnel to a non-allowlisted far end drops exactly like a `curl` to it. Tunneling to an arbitrary/ephemeral host degrades into a per-endpoint deliberation.
- **Phase-split** (from Codex Cloud) — this preserves the maximized action space: a *setup phase* (user-authored install/clone scripts, model not yet in the loop → no prompt-injection surface; supply-chain exposure remains, and is the lockfile's and the vendor-trust open decision's plane — precision added 2026-07-07) runs with the broad dependency preset or full egress; the *action phase* (model-autonomous) drops to the registry-derived allowlist.
- **Recorded residual risk**: no payload/content inspection (all three reference systems agree by omission) — GET-string exfil to an allowlisted host survives. This egress control converts ungoverned high-bandwidth exfil into governed, hardened, deliberation-gated egress; it is **not a DLP guarantee**. Deferred: content inspection, per-request approval, full DNS-rebind defense (clarified 2026-07-08, kairos prior-art review B1 — "full" scopes the deferral: the single-request resolve-once-and-pin transport itself is designed at the backend layer with a located donor; redirects and long-lived connections are the deferred remainder).

### Work-credential hardening — DEFERRED (added 2026-07-07, credential-isolation study)

The work-repo token above is Tier-0 contained (repo-scoped + physically separated from the Body remote + egress-bounded), not hidden. Three hardening candidates are designed-and-deferred, with provenance (`docs/research/2026-07-07-credential-isolation-codex-deepagents-hermes.md`):

- **Kernel-side git execution** — push/fetch performed by a kernel service holding the token (Applier-style); the sandbox sees only the working tree, and the token never enters it at all. This is Codex Cloud's repo model (its backend-held App token is inference-grade evidence there, recorded as such) — and the one path that would make the retired "never touches" claim literally true for work repos.
- **Phase-split credential lifetime** — the token exists only during a kernel-run clone/setup step and is wiped before the agent loop starts (Codex Cloud's secrets model, first-hand confirmed); requires guarding against setup-phase laundering into workspace files (`~/.bashrc`).
- **Unconditional env strip-list at every spawn boundary, fail-closed** (Hermes pattern) — defense-in-depth for kernel spawns: by design the sandbox holds no env secrets; this guards that design against regression, and no skill or config may re-allow a provider credential.

**Revisit trigger:** shared with Tier 1/2 below, or the first time a work token grants access beyond repositories the user personally owns — whichever comes first.

### Body write-path hardening — DEFERRED (Tier 1/2, 2026-07-06)

The write-path integrity above is **Tier 0: plain access control** — the sandbox cannot write the Body, the Applier can. That closes the critical write-path bug and is sufficient while this is a **single-user, self-hosted system on trusted infrastructure**. Two cryptographic hardening tiers are *designed-and-deferred*, pulled forward only when the threat model actually changes:

- **Tier 1 — commit signing**: the Applier signs each commit with one machine-held key; a pre-receive hook rejects any unsigned/forged push. Guards against a leaked push path. One key, no ceremony.
- **Tier 2 — user-approval signatures + key lifecycle**: the user signs an approval token over the approval's packet content-hash (the same hash the approval event already records — see *Trust Roots & Principal Authentication*; aligned 2026-07-06, superseding this bullet's earlier "hash of the approved diff" wording), stapled in the commit and verified against a registered off-server/hardware-backed user public key; plus a keyring with validity windows, rotation, revocation, and operator break-glass. This is what would cryptographically authenticate the human approver (review §4.1) and give secrets a time dimension (review §4.4), and its content-hash binding would close the what-you-sign gap. Enterprise-grade separation-of-authority — **premature for a single user**.

Why deferred, plainly: keys buy defense that survives a breach of the access-control layer, plus cross-principal non-repudiation — a single-user self-hosted system has neither threat yet, so the key ceremony would be complexity without a matching adversary. **Revisit trigger:** first hosted/multi-user deployment, any move onto infrastructure the user does not control, or the Operator and User hats splitting into different people (Axiom O's trigger — aligned 2026-07-06; two people make cross-principal non-repudiation a real need). Full mechanics (custody, keyring, rotation authority, compromise response) are drafted in `docs/reviews/2026-07-06-patch-body-write-integrity.md` for when this is picked up.

## Trust Roots & Principal Authentication (added 2026-07-06, resolves review §4.1)

Every gate in this document verifies an act against an authority — the Applier checks user approval, the extraction gate checks a directive's target — but nothing said how a principal is *known*, what an approval physically *is*, or what makes the records these checks read trustworthy. That was the review's last critical finding: the trust chain bottoms out in roots the document used without naming. At the current stage — **single-user, self-hosted, on infrastructure the user controls** — the resolution is not machinery: name the roots as axioms, adopt the few mechanisms whose adversary already exists at this stage, and record everything else as accepted risk with a revisit trigger. Nothing here re-opens the Tier 1/2 deferral above; this section defines the objects those tiers will later sign.

**Two axioms and one named non-root (decided 2026-07-06).**

- **Axiom H — the host.** The machine and OS account the kernel runs on are trusted absolutely: that root holds the session log, the vault at rest, the Applier's write path, and every gate's code. The root is wider than the login — anyone with write access to the stores' storage (a NAS mount, a backup credential) and anything living inside the user's browser profile (a malicious extension sits *inside* the approval surface's origin and defeats its checks) is host-equivalent. No in-system mechanism defends against this root and none below pretends to; defense here is machine hygiene (OS account control, disk encryption, extension discipline), outside this document's scope. Accepted risk: host compromise is total and silent. Revisit trigger: shared with Tier 1/2.
- **Axiom O — the human at the account.** One human currently holds both hats, Operator and User; principal authentication at this stage *is* possession of the host OS session — anyone at the unlocked keyboard is that human, by axiom, not by proof. What keeps the axiom honest is that the two hats still act through **two distinct channels** — the operator channel (kernel releases, Sonia's definition) and the user channel (approvals, instructions, charter writes) — each auditable on its own, so the multi-party upgrade is a role split (bind each existing channel to its own authenticated principal; Tier 2) rather than a redesign. Accepted risk: nothing in-system can prove which hat acted. Revisit trigger: Operator and User become different people.
- **Named non-root — vendor & infrastructure trust.** The model API inside every decision, upstream registries and code beyond what the lockfile pins, the OS and hardware vendors: a real trust plane this section deliberately does not axiomatize and does not govern. **Open decision (deferred)**, tracked in the reviews ledger as its own item — it is the blindspot scan's "vendor absent from the threat model" gap (distinct from review §4.3, which is abuse economics) — so the absence is a decision, not silence.

**What "user approval" is (decided 2026-07-06, resolves review §4.1).** With signatures deferred, an approval is a **kernel-appended event**: the user acts on the kernel's approval surface, and the kernel appends to the deliberation/audit log an approval event recording **deliberation-ID × the content-hash of the kernel-rendered packet** — the hash covers the **whole rendered artifact**: the candidate diff, every machine-generated field (behavior diff with its replay-fidelity declaration, non-behavior delta declaration, per-refinement cost, rollback point), the directive quote if any, and Sonia's argument text; hashing the full packet costs nothing and leaves no unpinned remainder. At landing the Applier verifies both bindings: the approval names the packet hash, and the packet pins the diff — an approval cannot be replayed against a different packet, and what lands is what was shown. This is Tier 2's content-hash discipline adopted **without its keys**: when Tier 2 arrives, the user's key signs this same packet hash — the upgrade changes what *authenticates* the approval event, never what it *is*. Honest limit, recorded: the hash pins bytes, not comprehension — the packet's persuasive argument remains Sonia-authored, and she reads possibly-poisoned traces; the kernel-generated evidence fields and the raw channel remain the defenses on that front.

**The approval surface.** Its *form* is deliberately undecided (a CLI prompt, a native window, a locally-served page); what is decided (2026-07-06) is its **admission rule**: the surface must be reachable only from the user's own OS session, and any form reachable by a browser must treat the browser as hostile — Origin **and** Host validated against expected literals on every request, reads included (DNS rebinding exfiltrates as well as forges), **fail-closed** on absent or mismatched headers; an unguessable per-session token never carried in a URL; no state-changing GET. Loopback binding alone is not an authentication boundary: any malicious page in the user's own browser reaches loopback via CSRF or DNS rebinding — a today-adversary that needs no second human (prior art: `docs/research/2026-07-02-sandbox-design-kairos-sonia.zh.md` §6). The surface and its channel-possession checks are kernel mechanism (enumerated with the packet renderer). Generalized 2026-07-07 (lessons-input-modes review): this admission rule governs **every** user-channel entry surface, the lessons dialogue surface included — a forged cross-origin request there could otherwise mint citable directives — and is pinned as a boundary property of Sonia's kernel, with a drill target (*Operations → Guard machinery is exercised*).

**Event-level principal origin (decided 2026-07-06 — the verbatim-quote fix, review top-5 item 4; the substrate review §4.2 will extend).** Every session event is stamped at intake — by `emitEvent()`, from the physical entry path, never inferred from content — with its origin channel: **user-channel** (entered through a kernel-served surface only the user's OS session reaches: conversational turns to either agent, approval acts, charter/correction writes, direct Body edits — added 2026-07-08) / **tool-result** / **model** / **kernel** (version events, checkpoints, watchdog reports). Consumption rule, with a named executor: a directive-origin proposal must quote a user-channel-stamped event, and the **packet renderer** — kernel code — verifies two things before rendering a directive quote: the quoted text is byte-identical to the referenced event's recorded text, and that event carries the user-channel stamp; anything else is refused; the memory-content taint tag is copied from this same stamp (see *Memory writes — one path*). Within the curated corpus (the evidence-curation gate, 2026-07-07 — see *Sonia: The Teacher Agent*), Sonia reads tool-result text freely — it is her evidence; it just can never be *cited as the user's instruction*. Honest limit, recorded: the stamp authenticates channel of entry, not authorship. It **narrows** the verbatim-quote laundering — injected tool-result text can no longer be quoted as a directive — but the user-as-courier edge survives: text the user pastes or is persuaded to type acquires a legitimate user-channel stamp, and a bare "yes, do that" ratifying injected text is user-channel while its referent is not; the extraction gate's scope classification and the charter citation are the only mitigations there. Sonia's kernel, when designed, must implement this same intake contract (stamp + chain below) — pinned now as one of her boundary properties. An external A2A counterparty is the fifth origin value, `a2a-party` (renamed 2026-07-07 from `served-party`) — non-citable as a directive, never a sole fact source, lower-trust than tool-result; cashed in 2026-07-06 as an extension, not a redesign (resolves review §4.2; see *The External Channel*).

**User-channel splits by physical entry path: authored vs couriered (decided 2026-07-07, lessons-input-modes review).** Pasting a link, uploading a file, or pasting a block of third-party text is a different physical act from typing an instruction, and the stamp records it — `user-channel/authored` vs `user-channel/couriered`, assigned at intake from the entry action, never inferred from content. Consumption rules: couriered material is **never directive-citable** (the packet renderer refuses it on the same code path that refuses non-user-channel quotes) and **never a sole fact source** (extending the tool-result rule — see *Memory writes — one path*); every couriered item carries a mandatory linkback to its ingested artifact (ingestion-at-the-gate — see *Sonia: The Teacher Agent*). Honest limit, recorded: a short inline paste is mechanically indistinguishable from typing, and persuaded-to-type text still acquires an authored stamp — the user-as-courier residual narrows but survives; its adversary is whoever authors what the user pastes.

**Cross-log citation resolution (decided 2026-07-07, lessons-input-modes review).** Directives live in *lessons* sessions — Sonia's log — while the packet renderer is Kairos-side kernel machinery; without a defined read path, the directive-citation check was unexecutable against its main source. Decided: event references are globally namespaced (plane × session × event); the renderer gains a **kernel-mediated read-only leg** into the lessons log, used solely for byte-identity + stamp verification of quoted events; and the user's raw channel (*Evolution Deliberation Channel*) extends to the lessons log. No write path crosses — the gate-complex write topology is unchanged.

**Record integrity (decided 2026-07-06 — detection, not proof).** The session log **and** the deliberation/audit log (the store holding approvals, dispositions, and extraction labels) both get an **append-time hash chain**: each appended record carries its predecessor's hash, and checkpoints and the recovery tuple record the current **chain head**. Verification is incremental at append, head-checked at `wake()` / checkpoint / restore, with a full walk only in the restore-test drill (see *Operations → Durability*). Scope stated exactly: the chain detects storage-side corruption, loss, or reorder of already-appended records, and interior edits by anything below the host root; it does **not** detect an emit-side bug (an event never emitted leaves a valid chain), tail truncation beyond the last recorded head, or the host itself (rewrite-and-recompute — Axiom H). Cost: one hash per append. External anchoring, WORM, signed chain heads: deferred with Tier 2 — what they buy (tamper-evidence of *pre-compromise* history against a later host compromise) is judged not worth the ceremony while the host is the accepted root.

**Kernel releases (decided 2026-07-06).** The release channel's integrity at this stage *is* Axiom O. Operations already stamps a version event and canaries each release; the version event now also carries the release's identity and a changelog/source reference, so "what changed" is answerable in-system. Recorded honestly: that record is written by the component under suspicion — it aids honest-but-buggy releases; against a *malicious* release the blast radius is total and in-system-invisible (a poisoned kernel forges origin stamps, re-links chains, forges approvals, and passes its own canary) — Axiom O by another door. The guard machinery this section adds joins the Operations drill roster (see *Guard machinery is exercised*). Deferred, with the adversary named: release signing, reproducible builds, two-person control — their adversary is an untrusted release *path*, which does not exist while the operator builds and deploys from the machine Axiom H covers. Revisit trigger: the first release built or delivered by anything other than the operator's own machine (a CI pipeline, a third party), or the Tier-1/2 trigger, whichever comes first.

## The External Channel: A2A Counterparties (rewritten 2026-07-07; supersedes *The Served Interlocutor: An Untrusted External Party*, added 2026-07-06, resolves review §4.2 — its five-row threat surface moves to the ROADMAP as deferred adversarial assumptions; full text in git history)

Kairos is pointed outward through an **A2A / agent-internet platform**: it exposes an A2A endpoint (agent card; task/message/artifact semantics) and serves **external counterparties** — peer agents, or humans behind them. This is the architecture's intended deployment shape, not a contingency (reframed 2026-07-07; the superseded section treated outward service as hypothetical). Trust Roots authenticated the *authority* side of the chain; this section names the other side. The insight the superseded section recorded stands: the roster folded "the party Kairos talks to" and "the authority who approves changes" into one benign **User** — pointed outward, those are two different parties, and the first is outside the trust circle. Two directions, kept distinct:

- **Inbound** — a counterparty calls Kairos. The counterparty is *outside* the four-principal trust circle: zero authority — never approves, never proposes, never writes memory; its action space is exactly the protocol's message set (send messages/tasks, supply files, induce tool outcomes, receive Kairos's replies); its every input is evidence at most. Its events are stamped **`a2a-party`** at the protocol intake (renamed 2026-07-07 from `served-party`; same emit-time, physical-entry-path, never-inferred-from-content contract — *Trust Roots*), stamped only once the surface exists (instance-less today). Consumption rules unchanged: never citable as a directive, never a sole fact source, lower-trust than tool-result, taints any generalization drawn from it.
- **Outbound** — Kairos calls another agent on the platform. That peer is a **connector**: it enters through the egress registry's single governance surface like any other connector (*Security Boundary → Sandbox egress*), and its responses are tool-result-origin. The two directions never blur: an inbound party is not an outbound dependency.

**Why this section exists before the first counterparty:** classification must precede mixing — the labels are nearly free now and cannot be retrofitted onto learning already mixed. The scope field {**agent-global** / **per-party** (renamed 2026-07-07 from per-user — the old value collided with the principal **User**, against one-name-per-write-regime) / **per-session**} rides every learned-context write, every deliberation-landed Body edit, and the charter; the approval inherits the packet's scope through the packet hash (*Trust Roots*). The kernel's **scope-mismatch check** is live from day one (decided 2026-07-06, resolves review §4.2): any edit landing at a scope wider than its cited evidence's scope fails the static policy gate and bounces to Sonia — never authored by the proposer, vacuous under a single User, un-retrofittable if skipped, and invisible on a behavior diff (a widened edit and a correctly-scoped one replay identically). Evidence scope is computed from the **induced-by principal-origin attribution** of the cited events and signals, not from a signal event's own kernel stamp (pinned 2026-07-07: a party-induced failure signature is kernel-observed yet party-originated — reading only the surface stamp would launder an adversary's influence into trusted evidence; see *Resources as Security*).

**The evidence-curation gate is the membrane (decided 2026-07-07):** only User-fed material enters Sonia's view (see *Sonia: The Teacher Agent*), so no counterparty text reaches evolution except through the User's deliberate selection — and selection grants visibility, never authority: a fed session's `a2a-party` and tool-result stamps survive verbatim.

**Platform trust:** the A2A platform itself is third-party infrastructure — one more face of the *vendor & infrastructure trust* open decision (*Trust Roots*), deliberately not axiomatized here.

**Deferred — adversarial assumptions and tenant machinery (moved to ROADMAP 2026-07-07).** The superseded section's five-row threat surface (fake corrections, poisoned ingested content, taste baked agent-global, cross-party confidentiality leak, induced failure signatures) is deferred to the ROADMAP as one block with the tenant machinery (per-party stores and retrieval isolation, party authentication, per-party conflict filters, cross-party corroboration counting, per-party rate limiting, Philosophy-B personalization). What the charter retains is what those rows decided: the origin value, the scope field and scope-mismatch check, the induced-by attribution rule above, and the signal/content separation (*Refinement Triggers*). **Revisit trigger: when Kairos first serves any counterparty other than the User** — the event that first splits *the party Kairos talks to* from *the User who approves* (corrected 2026-07-07: this is distinct from Axiom O's Operator/User split; either can occur without the other). Accepted risk, recorded: while single-user, agent-global promotion rests on one human's cross-session evidence; the charter and the raw channel remain the checks on that.

(The superseded section's threat-surface rows and decision bullets — including the scope-mismatch check's full mechanics, which remain in force via the *Evolution Deliberation Channel* packet contents and the Immutable Kernel enumeration — were absorbed above and into the ROADMAP's deferred-adversarial-assumptions block, 2026-07-07; full text in git history.)

## Resources as Security: Cost Is Also the Adversary's Weapon (added 2026-07-06, resolves review §4.3)

Cost enters this document only as a virtue — the per-refinement scalar on every packet (*Deliberation packet*), the kernel-derived counter on every component (*Per-component telemetry*), "the only objectively measurable scalar." But a scalar that is only *reported* is not *governed*: nothing sets a threshold on it, and the watchdog watches **failure**, not **spend**. So the measurable virtue is also the weapon — and unlike *The External Channel*'s deferred adversaries, which need a counterparty this deployment does not yet have, **both of §4.3's adversaries are live today** on the single-user machine: a prompt-injected activity session (injection arrives through the tool-results and web content Kairos reads — *Security Boundary → Sandbox egress*) and a buggy or looping Body can each burn a four-digit bill with every call succeeding and no failure signature firing, right now. This section adds the money dimension the compute quotas never had; it is a new watched signal under the authority that already exists, not a second authority.

**The threat surface** (each row: what the adversary does · what already blunts it · what this section adds):

- *Amplification* — injected content or a low-privilege interlocutor stages many, long, expensive-to-replay sessions; thereafter every packet's fork replay re-pays that inference cost over the recorded session, and every dreaming pass reads it too — one cheap act taxes all later evolution. **Blunted:** replay has no live egress (*Trial-run execution semantics*), so the amplified cost is model-inference compute, not re-sent external calls. **Added:** a spend ceiling on each replay batch and dreaming pass **bounds** (not eliminates) the amplified bill. (Blunting extended 2026-07-07: under the evidence-curation gate, dreaming and packet replay read only User-fed sessions — a staged corpus taxes later evolution only if the User feeds it.)
- *Economic denial-of-service* — every call succeeds, no failure signature fires, the user wakes to a four-digit bill. **Blunted:** nothing today — this is the true gap. **Added:** the spend budget makes spend itself an objective signal the watchdog enforces beside its failure signatures — the spine of the section.
- *Watchdog-trigger induction* — the failure triggers (loops, repeated tool failures, stalls — *Refinement Triggers*) are attacker-influenceable, so an adversary induces the signatures the watchdog watches to weaponize the kill-switch. **Blunted:** the kill is bounded to the *offending* subtree (*Machine authority boundary*), so an induced kill cannot cascade to other sessions or the whole instance — blast-radius limiting is the real protection. **Added:** each failure signature carries the principal-origin attribution of what induced it (*Trust Roots → Event-level principal origin*), so repeated induced kills traceable to injected or a2a-party content become a recorded signal, not a free anonymous DoS. **Honest limit (recorded below):** today's live injection rides *inside the user's own session*, so a bounded kill can still land on the user's own work — bounded, not deflected onto "the attacker's session."
- *Silent price drift* — "the only objectively measurable scalar" is silently indexed to a vendor price list the system cannot see change. **Added:** the price list becomes a recorded, versioned kernel-hosted input, so money is a real unit and a *pin update* is a visible deliberation (below).

**Decisions.**

- **The spend budget and spend-watchdog — money is a first-class watched signal (decided 2026-07-06, resolves review §4.3).** The orchestration compute quotas (*Immutable Kernel* — spawn budget, recursion depth, concurrency caps) gain a **money axis**: a money-denominated **spend ceiling** at each scope — per activity session/subtree, per calendar day (cross-session, a kernel-written accumulator row — *Immutable Kernel*, observability/ledger class), per dreaming pass, per replay batch. The **watched signal is the kernel's own metering** of every model call it dispatches (token count × price) and every egress call through the vault/egress proxy — not merely the per-component tool counter (*Per-component telemetry*), so a **pure-inference loop with no tool calls is still metered**, which is exactly the all-success runaway that fires no failure signature. Enforcement action is keyed to scope and attendance, because one instance-wide kill would itself be the DoS §4.3 warns of (it would take the user's concurrent work): an **unattended** session/subtree that breaches its ceiling is stopped by the existing bounded detect-and-kill (*Machine authority boundary* — one offending subtree, no Body substituted); a **foreground interactive** session that breaches **pauses and prompts** the present user (re-authorize or kill) — the work may carry injected content, but a present user can adjudicate, so a pause beats a hard kill of live work (reworded 2026-07-07); and the **cross-session daily** ceiling, which has no single offending subtree to kill, **stops admitting new spend/new sessions and pages the user** rather than killing work in flight. One enforcer (the watchdog), no FinOps platform, no soft/hard tiering of a single ceiling — different scopes genuinely need different actions. The ceiling **values** are user-direct records (classed 2026-07-07): kernel-side, user-owned, writable by neither agent — the same write regime as the charter and correction records; the budget is the user's own tolerance declaration, and no deliberation stands between the user and their wallet, while the metering and enforcement code stays kernel mechanism. **Ceiling-value edit semantics — OPEN DECISION, deferred on re-review (2026-07-08):** whether a user-direct ceiling-value edit affects in-flight sessions is undecided — immediate-effect and admission-time snapshot binding (kairos `admit_run`) were both evaluated, neither adopted; until decided, faces display ceilings without asserting propagation semantics.
- **Money is denominated by a versioned price list (decided 2026-07-06, resolves review §4.3).** Spend is money only against a rate. A **price list** — per-model-id rates (the attribution tuple already carries model-id, so a model swap re-prices correctly) and per-egress-call rates — is folded into the **third-party lockfile** as a versioned, kernel-hosted governed data row (agent-unwritable), and a price change rides the **expedited lockfile lane** (*Operations → the outside world changes on its own clock*). Honest limit: there is no external price feed, so a **silent vendor repricing is caught only when the pin is reconciled**, not in the moment it takes effect; what is visible is the pin-update deliberation. This touches the "vendor & infrastructure trust" Open decision (*Trust Roots*) at the one point money cannot avoid it — the price list — without axiomatizing the rest of that plane.
- **Anti-induction: bounded scope is the protection; attribution makes repetition a signal (decided 2026-07-06, resolves review §4.3).** The primary defense against trigger-induction is structural and already present: the kill is scoped to the offending subtree (*Machine authority boundary*), so inducing it cannot cascade beyond that one work unit. Attribution hardens the margin: every failure signature that drives a kill or an out-of-band proposal records the principal-origin of the events that induced it, so a pattern of kills induced by injected tool-result or a2a-party content is reviewable rather than free-and-anonymous. The *spend* trigger's per-subtree kill is bounded the same way — inducing it kills the offending subtree, not the instance (the cross-session daily ceiling's halt-admission is the one exception, recorded as an availability residual below). **Honest limit, recorded:** attribution is imperfect (the user-as-courier and injected-tool-result edges survive exactly as in *Trust Roots*), and because today's injection rides inside the user's own session, a bounded kill can land on the user's own work — this bounds the weaponization, it does not eliminate it.
- **Symmetric to Sonia's plane (decided 2026-07-06, resolves review §4.3).** The money axis is not Kairos-only. Sonia's lessons plane already carries the objective-stratum floor (*Machine authority is symmetric*); it now also carries the spend ceilings — dreaming passes and trial-run replay batches are precisely where amplified cost lands, so an unattended overnight consolidation or replay loop is killed on **spend** exactly as on stall or thrash, its incidents filing to the operator channel. **Recorded residual:** halting the lessons plane at its ceiling is itself an availability lever — an attacker who stages expensive-to-replay sessions can trip Sonia's ceiling and stall evolution; per-session replay skip/sampling is a possible future refinement, not built.
- **Deferred — per-party rate limiting, with the §4.2 trigger (decided 2026-07-06, resolves review §4.3).** A *per-interlocutor* rate or spend limit — throttling one counterparty's session-creation so it cannot stage the amplification corpus at all — genuinely needs counterparties to limit; building per-party limiters with no parties is the rejected over-build. The system-wide ceilings above already cap the bill today regardless of who induced it; only the per-party split defers, as one block with *The External Channel*'s tenant machinery. **Revisit trigger: when Kairos first serves any interlocutor other than the User.**

**Accepted risks, recorded.** (1) *The attack's own bill up to the cap is still paid* — the ceiling bounds the loss, it does not refund it; one budget period's induced spend is the accepted floor. (2) *Granularity is one call* — the signal is checked between calls, so a single call larger than the remaining budget (up to the concurrency cap's worth in flight) overshoots by that much; sub-call metering is not built. (3) *A wrong or stale price list makes the ceiling wrong* — the budget is only as true as the pinned prices, until the expedited-lane deliberation reconciles them. (4) *Foreground disruption / collateral kill* — a pause of the session the user is actively using stops real work (and an unattended subtree carrying injected content is killed outright); what makes it acceptable is that a present user can re-authorize a foreground pause, while the unattended planes — where an all-success runaway is genuinely dangerous — have no one to interrupt and so hard-halt. (5) *Daily-ceiling availability lever* — tripping the cross-session daily ceiling halts new-session admission instance-wide, so an attacker who stages spend toward the cap can deny the user the ability to start new work (the same availability lever recorded for Sonia's lessons plane above); nothing throttles session-creation rate today — the mitigation, per-party rate limiting, is deferred with the §4.2 trigger.

## Session Is Not the Context Window

Another strong design decision: the session (event log) and the model's context window are separate things.

Long-running tasks exceed the capacity of the context window. Traditional approaches such as compression, summarization, and trimming all involve irreversible tradeoffs, and it is hard to predict which tokens will matter later. This design (following CMA — charter voice 2026-07-07) instead keeps the complete conversation history in the session log forever. The loop can call `getEvents()` at any time to look up any historical fragment. Before sending events into the context window, the loop can transform them however it wants: sorting, filtering, reorganizing, depending on the current need.

The benefit is that recoverable context storage (the session's responsibility) is separated from arbitrary context engineering (the loop's responsibility). The design does not need to predict what context management strategy future models will need: as long as the raw record is always available, the strategy can change at any time.

The mirror principle on the other side: the context window is not a component at all — its content belongs to Body and Session; only its assembly logic belongs to the kernel (see *Harness = Kernel + Body*).

**Traces — DEFERRED.** Trace design (schema, stamps, storage) is deferred; nothing in this document may presuppose it is built. Recorded leaning for when it is picked up: the session event log stays the single source of truth — the stamps evolution needs (the attribution tuple at event: body-version × model-id × kernel-version; per-step cost; workflow-instance-ID lineage; correction marks) live as event metadata on this same append-only stream; late-arriving marks are appended annotation events referencing prior event IDs, never in-place mutation; a trace = a structured read view over the log, not a second logging system. Four pieces are carved out as **non-deferred** because decided mechanisms already depend on them (2026-07-06): the attribution-tuple stamp (watchdog incident attribution — see *Edit Acceptance Protocol*), the kernel counter-event schema (see *Component Lifecycle & Telemetry*), and the principal-origin stamp and append-time integrity chain (render-time citation checks and wake-time verification depend on them — see *Trust Roots & Principal Authentication*); everything richer stays deferred. Either way, the event log remains the system's most valuable data asset.

**Log lifecycle (decided 2026-07-06, resolves review §2.8).** "Append-only forever" owes four mechanics — three against time, one against corruption: (1) **schema versioning** — every event carries a schema-version envelope, and each kernel release ships read-time upcasters for older versions; history is never migrated in place, it is reinterpreted at read time (a `wake()` that crosses two kernel releases must still parse its genesis events); (2) **checkpoints** — the kernel periodically appends derived checkpoint events, so `wake(sessionId)` resumes from the latest checkpoint instead of replaying from genesis and recovery time stops growing with session age (a checkpoint is a derived, appended view — nothing edits the log); (3) **tiering** — cold segments move to cheaper storage under the same append-only guarantee, with `getEvents()` reading through transparently, so "keep everything forever" has a cost story on the hot path: fork replays and packet evidence pay for recent segments, not for the archive; (4) **integrity** (added 2026-07-06) — every appended record carries its predecessor's hash, checkpoints and the recovery tuple record the chain head, and verification is incremental at append with a full walk only in the restore-test drill — see *Trust Roots & Principal Authentication* for exactly what this does and does not detect.

## Many Brains, Many Hands: Horizontal Scaling

Decoupling directly enables horizontal scaling.

- **Many brains**: In the pre-split architecture, each session required starting a container, and inference had to wait for the container to finish booting. After the split, the loop is stateless, so sessions that do not need a container do not need to wait.
- **Many hands**: Each brain can connect to multiple sandboxes because each hand is just a tool interface. The loop does not care whether the sandbox is a container, a phone, or (CMA's own example) a Pokémon emulator. Since hands are not bound to brains, brains can pass hand ownership to each other.



## Built for Long-Running, Stateful Work

Sessions run over WebSockets with streamed output, and state persists across threads; sandboxes stay alive only as long as needed (vendor voice reworded 2026-07-07).

- **Configurable TTLs**: inactive TTLs shut idle sandboxes down automatically; hard TTLs cap total runtime.
- **Snapshot and fork**: a running sandbox can be snapshotted to capture its full state, restored later to resume, or forked into multiple sandboxes so an agent can branch and explore alternatives in parallel. Fork doubles as the evolution layer's instrument (added 2026-07-04): trial-running a proposed Body against replayed sessions to generate the deliberation packet's behavior diff, and holding the pre-edit state as rollback insurance — not merely task exploration.
- **Port tunneling**: any port inside the sandbox can be tunneled out to preview a running app, hit an HTTP endpoint, or connect to a service the agent spun up. The tunnel's outbound leg is governed by the egress allowlist — a tunnel to a non-allowlisted far end drops exactly like a `curl` to it (cross-ref added 2026-07-07; see *Security Boundary → Sandbox egress*).



## Memory Design: Two Philosophies

Two production systems answered "how should an agent remember" in opposite ways during 2026: Claude Managed Agents' memory stores (file-based) and Mem0 (extraction-pipeline). They are not competing implementations of one idea — they are different bets about where the intelligence lives.

### Philosophy A — Memory as Workspace (file-based; Claude Managed Agents)

Memories are plain text files mounted into the sandbox; the agent reads and writes them with the same file tools it uses for everything else. No vector index, no embeddings, no semantic search: retrieval is the model navigating a directory tree, guided only by a mount note injected into the system prompt. The bet is on **model intelligence** — a model trained for filesystem memory decides what to remember and what to look up. The core idea is Claude Code's `CLAUDE.md` and memory files, restructured as an API (Lance Martin's framing).

Design details worth keeping:

- `read_only` / `read_write` enforced at the filesystem level — a *structural* prompt-injection defense: a poisoned session cannot write into a read-only store, no matter what the model is convinced to do.
- Every modification creates an immutable, session-attributed version record (audit, traceability, point-in-time recovery).
- Up to 8 stores per session, and hard caps (100 kB per memory, 2,000 memories per store) act as deliberate forcing functions toward small, well-named, greppable files instead of sprawling journals.
- Consolidation ("Dreams") reads the store plus past sessions and outputs a **new** store for human review; the input is never mutated.



### Philosophy B — Memory as a Service (extraction-pipeline; Mem0)

An LLM distills each exchange into small self-contained facts stored in a vector store; before the next model call, a single-pass hybrid search (semantic + keyword + entity) injects a few thousand tokens of relevant memories instead of replaying history. The bet is on **infrastructure intelligence** — model-agnostic, sub-second retrieval, built for per-end-user personalization at scale. The costs: extraction is lossy (what is stored is a machine transformation of what was said), and superseded facts are never removed — they must lose at ranking time.

Instructive: Mem0's April 2026 v3 rewrite *abandoned* LLM-arbitrated memory updates (ADD/UPDATE/DELETE decisions at write time) for a one-call, ADD-only pipeline. A year of production traffic taught them that write-time intelligence is not worth its cost and risk — intelligence moved to retrieval time.

### The Convergence Principle: Never Auto-Destroy

Both systems, starting from opposite ends, landed on the same rule: **memory is never destructively modified by an automatic process.** CMA versions every mutation immutably and consolidates into fresh stores; Mem0 went ADD-only and resolves conflicts by ranking. This is "Session Is Not the Context Window" one layer up: irreversible information loss is the failure mode to design against. Push conflict resolution to read time or to human review — never into an in-place automatic write.

Body-layer applicability (decided 2026-07-04): the principle extends to executable components (skills, sub-agents, workflows) as **archive-not-delete** — a third state outside retrieval and loading but inside audit history — with one exemption: credentials and OAuth grants are hard-deletable, because revocation must actually revoke. See *Component Lifecycle & Telemetry* under the Evolution Layer.

### Decision for SoniaKairos

Layered, not either/or:

- **Agent working memory** (project state, engineering notes, learned facts): **store of record is Mem0** (decided 2026-07-09 — supersedes the earlier file-based Philosophy-A choice; see *Substrate* below). This adopts Philosophy B's substrate (vector store + retrieval) for working memory, but under this design's governance and with its two main Philosophy-B liabilities designed out: **fidelity** — approved content is stored `infer=False` (verbatim), so the lossy extraction that makes Mem0 an "opinion engine" is confined to Sonia's *drafting* and never touches stored text; **tamper resistance** — Kairos's runtime holds a retrieval-only Mem0 client with no write path (process/network isolation), the structural successor to the filesystem `read_only` mount. Since the 2026-07-06 pivot this memory is **written by Sonia as deliberation proposals — or (2026-07-08) by the User directly through the Applier — never by Kairos**, and is **typed** — facts/state only; procedures live in Sonia-authored skills/workflows, never as memory content. The hand-curated `CLAUDE.md`/skills layer is the read-only end of this same spectrum. Post-pivot invariant (recorded 2026-07-07, mechanism updated 2026-07-09): the activity plane is read-only against memory — the Applier is the sole writer (now via Mem0's write API), and trial-run forks get a read-only retrieval view — so the read-only defense is a global invariant of the activity plane, not a per-store setting; Mem0's `read_write` client has no Kairos-side consumer.
- **End-user personalization** (when Kairos serves external counterparties — the intended A2A shape, not a contingency; reworded 2026-07-07, see *The External Channel*): **extraction-pipeline, Philosophy B** — per-party facts (per-end-user, in Mem0's vocabulary) behind scoped IDs, retrieval kept off the response critical path. This layer, with per-party stores and retrieval isolation, is the **deferred tenant machinery** — revisit trigger: when Kairos first serves any counterparty other than the User.
- Both layers obey the convergence principle: no automatic in-place destruction. Consolidation is a *proposal* (a new store / new revision) that is reviewed before it replaces the old.
- **Scope labels from day one (decided 2026-07-04; extended past the memory edge 2026-07-06, resolves review §4.2)**: every learned-context write — memory entries, Sonia-authored skills, extracted preferences, **and (2026-07-06) every deliberation-landed Body edit + the charter, with the approval inheriting the packet's scope** — carries a scope field: **agent-global** (Kairos's craft) / **per-party** (renamed 2026-07-07 from per-user; per-interlocutor — the single User today, an A2A counterparty when outward) / **per-session**. Single-user today, so the field is nearly free now; unlabeled accumulated learning can never be split later when a second user or instance arrives. Same logic as trace-eligibility: classification must precede mixing — now applied to the heaviest learned-context channel too (see *The External Channel*).
- **Substrate (decided 2026-07-09, user-ratified — supersedes the same-day assistant-default *machinery* reading):** memory's **store of record is Mem0 OSS**, not in-repo files. All reads and writes go through Mem0 — the Applier writes approved content with `add(infer=False)` (verbatim, no re-extraction: the exact text was already approved), Kairos reads through a retrieval-only client. What survives unchanged from the pivot: Kairos never writes memory, Sonia is the sole *agent* proposer, the Applier is the sole writer (it now calls Mem0's write API instead of committing a memory file), content is typed facts/state-only, and never-auto-destroy holds twice (Mem0 v3 is ADD-only; the journal below is append-only). **Rollback-coherence via event-sourcing:** every memory write is *also* an immutable row in the git-committed deliberation/audit journal (deliberation-ID or user-edit-ref FK — the log that already records every Body mutation). The git journal is the append-only **derivation** of the Mem0 store and is **reconcile-authoritative**: Mem0 persists durably and serves day-to-day, but on rollback or detected drift it is **rebuilt from the journal** up to the target commit — so "roll the Body back and memory comes with it" is a checkout + journal-replay, not a native Mem0 operation (source-verified against live `mem0ai==2.0.11`, 2026-07-09: no import/restore/checkout of the whole store exists in OSS — only per-memory ADD-only history + tombstone delete; hosted export is one-way/lossy/7-day-expiring and infra `pg_dump`/EBS snapshots are per-backend, coarse-cadence, and audit-unlinked — named and dismissed in Backend §5). "v3" throughout is the algorithm/TS-major label; the pinned Python package is `mem0ai==2.0.11`. This is the same *one-source-of-truth + idempotent reconciliation* pattern the vault leg uses (see *Cross-store consistency*); memory is a genuinely separate store again, but unlike the vault its **content is fully reconstructable from the journal** — retrieval *ranking* only to within retrieval-CI tolerance (must-surface items stay top-k), not bit-exact, because embeddings and ANN-index construction are not guaranteed deterministic and a from-empty replay reorders the write stream — so it is the strongest reconcile leg on content/durability, never a chimera risk. **read_only injection defense — honest limit:** Kairos's runtime holds a retrieval-only client, but this is *weaker* than the retired OS-enforced filesystem read-only mount — it is a client-level guarantee, and Kairos keeps network reach to Mem0 to read. To be structural the write surface must be credential-gated *and* unreachable to Kairos except via a read path: **Tier 0** — Kairos reaches only a read-only proxy, never the raw backend port (writes server-side, Applier-only); **Tier 1** — backend API-key auth (read-only key for Kairos, write key held solely by the Applier). Recorded limit: absent that, an injected same-host Kairos could write to a default-unauthenticated local backend (e.g. Qdrant) — see Backend §5 and *Deferred by decision*. **Fidelity:** the lossy-extraction objection to Philosophy B does not bite — storage is `infer=False` verbatim-approved text; Mem0's extraction pipeline is used only as Sonia's *drafting* tool (trace → candidate fact → deliberation packet), never at the storage boundary. **Carried forward from the study:** a Mem0 embedding/index change reshuffles retrieval silently (blindspot-map unversioned-index row), so the embedding-model id + index-build hash are pinned in the lockfile, a re-index is a declared version event, and the retrieval-CI probe suite runs on every landed write, on any re-index/embedding change, and on any pinned-Mem0 version bump. **Anti-lock-in:** because the journal is a portable append-only record, replacing Mem0 with any other store is a replay of the journal into it. **Superseded/rejected readings:** *machinery* (Mem0 as a throwaway index over file-based truth) — the same-day assistant-default, now superseded; *full-substrate-without-journal* (Mem0 as truth with no git derivation) — rejected: it forfeits rollback-coherence and a unified audit trail. Philosophy-B *per-party personalization* on Mem0's native ungoverned pattern remains the deferred tenant machinery above, unchanged. Donor detail, the not-used list, and accepted costs (zh-degradation, no rank-time temporal reasoning in OSS, no independent reproduction of v3, the retrieval-CI obligation, the widened erasure residual — the journal now also retains erased content): Backend-Design §5, Mem0 donor block; ROADMAP *Deferred by decision*.

## Advanced Features: What Only the Outer Loop Can Do



### Dreaming: Letting Agents Improve Between Sessions

Dreaming is a scheduled process that reviews **the User-curated corpus** (input domain narrowed 2026-07-07 by the evidence-curation gate — it does not autonomously scan the full log), extracts patterns, and organizes memory. Within the fed corpus it can find things a single agent would not see, such as recurring errors, shared team preferences, or workflows that agents converged on over time. Accepted cost, recorded (2026-07-07): pattern discovery is confined to what the User feeds; revisit if curation demonstrably starves evolution (merged with the offline-refinement-too-slow trigger, *First Founding Principle*).

The hosted platform offers a choice between automatic dreaming memory updates and human review; **for SoniaKairos the automatic-write option is disallowed** — dreaming produces deliberation proposals only, landed via Sonia → user approval → Applier (see *Memory writes — one path* and *Refinement Triggers*), never an ungated memory write. Dreaming also has an assigned principal and plane (decided 2026-07-06, resolves review §1.5): it is **Sonia's scheduled consolidation pass, run in lessons sessions** — a scheduled process that reads past sessions and calls a model is an agent by any name, so rather than becoming an unrostered third one, it runs as the principal whose job this already is. (Harvey, a legal AI company, reported ~6x agent task-completion improvement after adopting dreaming — a vendor claim with no baseline or task definition; treat as motivation, not evidence.)

### Outcomes: An Independent Evaluator

Outcomes takes a user-written rubric describing what success looks like, then asks an independent evaluation model to score the result in its own context window. The key point is that the evaluator is not influenced by the agent's reasoning process, because it looks at the output rather than the chain of thought. If the result fails, the evaluator points out concrete issues and the agent revises.

This separation between the person doing the work and the person evaluating the work matters in AI systems. (Vendor-published data claims outcomes improved task success rates by up to 10 percentage points over a standard prompt loop, with significant improvements in document generation quality — the same evidentiary status as Harvey's 6x above: vendor marketing with no baseline or task definition, kept as motivation, never load-bearing; 2026-07-06, resolves review §3.5.)

> **OPEN DECISION (2026-07-04)**: under this design's open-ended evaluation (no fixed benchmark; the standard is the user's shifting preference), a fixed rubric does not exist, and this evaluator can never be an acceptance authority — ACCEPT for evolution comes only from the user (see Edit Acceptance Protocol). Whether Outcomes survives as a demoted *advisory* anomaly channel is deferred. If it survives, its rubric/evaluator configuration belongs to the immutable kernel. Do not implement evolution acceptance against this section. **CLOSED (2026-07-07): not built.** The objective-stratum anomaly duty is carried by the kernel's signal rows (*Refinement Triggers*); at single-user scale the user reviews outcomes personally. Revisit trigger: when the user can no longer personally review output at scale (e.g., outward A2A service). The "never an acceptance authority" rule remains binding either way.

### Coordinator / Multi-Agent Orchestration

Multi-agent orchestration lets a main agent split work across specialized subagents. Each subagent can have its own model, prompt, and tools. They work in parallel on a shared file system, and the main agent can inspect other agents' progress along the way. (Netflix's platform team reportedly used this feature to analyze logs from hundreds of builds in parallel — a vendor case study, same non-load-bearing status as the other hosted-platform numbers in this chapter.)

> Editor's note: These advanced features, dreaming, outcomes, and multi-agent orchestration, are all built into the Managed Agents hosted service rather than exposed as general APIs. That makes sense: they all require outer-loop infrastructure, including the session log, scheduling, and independent evaluation inference. They cannot be solved by a plain model call alone.

### Immutable Kernel: The Trust Base of Evolution

Everything self-modification must *not* be able to touch, enforced structurally at the filesystem/permission level — the read_only memory-store pattern, one layer up. Binding on **both principals**: outside Kairos's self-edit reach *and* outside what an approved Sonia deliberation may change. Under *Harness = Kernel + Body*, this kernel is the harness's fixed part — the fixed function; it also hosts governed data rows, which are cargo, not mechanisms (see the mechanism/data note below the enumeration).

**Kernel enumeration** (never writable in-system):

- the kernel prompt stub — the always-present instruction segment: identity, security invariants, the deliberation-protocol instructions themselves. (Memory mount notes are *not* stub literals: the stub holds their template and delivery mechanism; their content is rendered per call from the Body repo's memory config — see *Harness = Kernel + Body*, fixed point 2.) The system prompt's other segment is Body and evolves; the stub does not
- runtime action & tool-call schema validation — every action is checked against the harness execution schema (a populated `tools_to_call` with the dispatch sentinel absent fails loud) and every dispatched call against the tool's declared parameter schema; malformed actions **fail loud**, never silently dropped (the actual Power Plant lesson — see *Edit Acceptance Protocol*); this is the only defense positioned to see call payloads, and it covers session-local execution no proposal gate ever reviews

**Mechanisms vs kernel-hosted data (clarified 2026-07-04)**: two strata inside this enumeration. *Mechanisms* — the loop and context assembly, stub delivery, vault/proxy, Applier + audit log + egress proxy, trace logger, quotas and kill-switch, the gate-checking code of registry/whitelist/lockfile — are the kernel proper, the harness's fixed part: operator release only, untouchable by any deliberation. *Kernel-hosted governed data rows* — egress registry entries, scope-whitelist entries, lockfile pins, connector kernel fields, user charter and correction records, the never-codifiable decision-class list, and the per-class **mutation grammars** (homed 2026-07-07; see *Edit semantics per component class*) — are agent-unwritable but change through their designated non-agent gates (deliberation → user approval → Applier; user-direct write for charter/corrections, spend ceilings, and the feed-list). A third row class (named 2026-07-06): ***kernel-written observability & ledger rows*** — per-component telemetry counters, kernel-service liveness telemetry, triage disposition entries, extraction-gate labels — appended through a kernel intake only (the ledger analogue of `emitEvent()`: an agent can *cause* an append through the gate, never edit or delete in place), so they are append-only and agent-unwritable without needing a deliberation per row. "Never writable in-system" above means: no agent write, ever; for mechanisms, additionally no deliberation.

**The Applier — the sole writer service (decided 2026-07-04; revised 2026-07-06)**: approved deliberations are landed by a deterministic, non-LLM applier service (a mechanism, not one of the four principals in the roster below): verify deliberation-ID + user approval (an approval is the kernel-appended approval event of *Trust Roots & Principal Authentication*: deliberation-ID × packet content-hash, checked against the diff being landed) → apply diff → commit → write an immutable, session-attributed audit record. **Sonia is the sole proposer; Kairos does not propose** (pivot 2026-07-06); neither can invoke the Applier's internals. Every persistent Body mutation carries a deliberation-ID foreign key or (amended 2026-07-08) a user-direct edit-event reference. **Second intake — the User's direct edit (decided 2026-07-08, re-review):** the User may modify the Body directly, without a deliberation: the edit enters through an admission-rule-guarded user surface, is stamped `user-channel/authored` at intake, and is landed by the **same Applier** — apply diff → commit → immutable audit record referencing the user-edit event. Nothing else changes hands: Kairos still never self-modifies, Sonia remains the sole *agent* proposer (a user-direct edit is not a proposal and never enters the queue), and the Applier remains the sole writer — this adds a second legitimate *trigger*, never a second writer. Versioning, rollback, and the raw channel cover user edits exactly as they cover landed deliberations, and Sonia reads landed user edits as evidence (they are part of the Body she proposes against) and may propose corrections. **Honest limits, recorded:** a user-direct edit forgoes the packet's kernel-generated counsel (behavior diff, scope-mismatch, dedup, coverage) — speed traded for evidence, knowingly; and the user-as-courier residual (*Trust Roots*) widens — a persuaded user can now land attacker-authored content directly, with no packet field in between; the bounds are the authored stamp, the audit trail, versioning/rollback, and Sonia's corrective read. **Write integrity is enforced by access control, not cryptography (Tier 0, decided 2026-07-06):** the Applier process is the only thing with write access to the Body remote; sandboxes mount it read-only and hold no push credential, so an injected activity session simply has no write path to poison the Body — the critical write-path fix needs nothing more at a single-user, self-hosted stage. Cryptographic hardening (commit-signing, a pre-receive hook, user-approval signatures, key lifecycle) is **deferred** — see *Body write-path hardening*. **Not merged into Sonia, or any proposer (decided 2026-07-06):** the Applier's value is that it *cannot do anything but land an approved diff* — a dumb deterministic executor, not a general agent. Sonia is an LLM agent reachable by prompt injection (she reads possibly-poisoned traces); giving the Body-write capability to the most-exposed principal would invert least-privilege and collapse the proposer/approver/executor separation of duties. An agent-applier could, *during* apply, do other things (touch memory, spawn sub-agents, reach the vault) — a dumb Applier structurally cannot. The Applier is a kernel-owned, Kairos-side neutral gate (see *Sonia: The Teacher Agent* — the pipeline crosses between the agents only through the gate complex it anchors: Applier, deliberation/audit log, packet renderer, trial-run evidence generators — plus, read-direction, the curation-gate filter and the renderer's lessons-log read leg, added 2026-07-07); merging it onto the proposer's side would let Sonia write directly into Kairos. Sonia *triggers* the Applier; it is never *part* of her. The Applier is the **sole writer of all Body content, memory included** — memory content lands here like any other Body edit (the old forward-to-write-waist step is retired along with the ungated waist). When one deliberation touches multiple stores (git repo, vault grants, memory), the git leg alone is not "applied" — multi-store crash-consistency is handled by commit-last apply + reconcile-to-tip (see *Cross-store consistency* under *Body Persistence*). A deterministic **never_relax hard gate** (kairos donor `constraint_gate`: security postures may only tighten, enforced before any write) was evaluated and **deferred (2026-07-09, re-review follow-up)** — every landing already passes a user act, and the kernel-generated non-behavior delta declaration flags relaxations to the sealing user; a hard monotonic gate would need enumerated exceptions for legitimate, deliberation-gated relaxations (egress-registry and scope-whitelist additions are by-design relaxations).

**Landing discipline (decided 2026-07-06, resolves review §2.10).** The Applier lands packets **serially**, and at landing verifies that the diff's parent commit equals both the current tip *and* the base the trial run replayed against. A stale or rebased packet bounces back to Sonia for re-derivation against the new tip — it is never auto-applied, because two packets cut from the same base, approved in sequence, would otherwise stack into a Body state no packet ever showed the user, invalidating both packets' evidence and rollback points. What the user approves is pinned the same way: the deliberation packet is **rendered by kernel code** (the presentation layer is a kernel mechanism, listed in the enumeration), and the packet's machine-generated fields — the behavior diff from fork replays, its replay-fidelity declaration, the non-behavior delta declaration from static analysis of the diff — are produced by kernel machinery, not authored by the proposer. Sonia argues *for* a change; the kernel states *what it is*.

**Memory writes — one path (decided 2026-07-04, collapsed 2026-07-06)**: the earlier design gave memory two write paths — ungated **content** via the write-waist, **structure** via deliberation — *because Kairos wrote notes on the hot path*. The pivot removed that writer: Kairos writes no memory. So content and structure now **share one path** — Sonia proposes → user approves → Applier lands, or (amended 2026-07-08) the User edits directly through the same Applier; one writer, two triggers — and the ungated write-waist is retired. **How the Applier lands memory (decided 2026-07-09, Mem0 substrate):** it writes the approved content to the Mem0 store of record via `add(infer=False)` (verbatim — no re-extraction of already-approved text) *and* commits the matching row to the git deliberation/audit journal, in the same commit-last apply; the journal row is the reconcile-authoritative record and the Mem0 write the serving copy (see *Memory Design → Substrate* and *Cross-store consistency → Memory leg*). Build-time contract (source-verified against pinned Mem0): the Applier passes approved content as a single `role="user"` message, because `add(infer=False)` role-filters before embedding and a non-user turn can be silently dropped — the sole write path must not lose data. Content is **typed**: facts/state become memory content; procedures/behavior are never memory content — they are Sonia-authored executable components (skills/workflows). This typing closes the old laundering hole (a note saying "next time do X" is behavior, not a fact) by construction. Accepted cost: the approval queue now carries routine facts; escape valve if it saturates (deferred) — an *expedited* Sonia-side facts-only channel that is still user-approved (lighter packet, batched approval), never an approval bypass; a channel that dropped user approval would reopen the governance axiom, not extend it. **Taint moved to the packet**: Sonia reads possibly-poisoned activity traces, so every memory-content proposal must carry source-trace linkback + a principal-origin tag in the origin stamp's vocabulary (user-channel authored|couriered / tool-result / model / kernel / a2a-party — vocabulary extended 2026-07-07), copied by the packet renderer from the kernel's emit-time stamp, never asserted by the proposer (decided 2026-07-06; see *Trust Roots & Principal Authentication*); a fact proposal may not rest solely on tool-result-origin text — nor on couriered text (extended 2026-07-07) — nor, outward-facing, on a2a-party-origin text, which is lower still (kernel-origin events — version events, watchdog reports — are admissible fact sources; see *The External Channel*).

**Self-modification — one form, one deferred (decided 2026-07-04; pivot 2026-07-06)**:

- *Persistent (the only form)*: Sonia proposes → user approval → Applier, or (amended 2026-07-08) the User's direct edit through the same Applier. No third path. Kairos does not propose and does not self-edit.
- *Session-local — **Deferred (2026-07-06)***: Kairos adapting its own prompt/skills/tools mid-session to unblock itself is shelved (see *First Founding Principle*). Until revisited, Kairos has **no authority over its own definition, ephemeral or persistent**. The Iron rule (nothing may auto-replay a session's ephemeral adaptations at boot) holds trivially, since there are no such adaptations. Accepted cost (recorded in the First Founding Principle): Kairos must fail into Sonia's offline refinement rather than self-unblock; the paper's ~500-turn-stall evidence is the accepted risk, to be re-weighed only from Kairos's own traces if the capability is revisited.
- Session taxonomy: Sonia's refinement of Kairos runs in its own sessions (**lessons**); Kairos's interactions with the external world run in separate **activity sessions**. Deliberation and audit live on the lessons plane; **User-curated** activity traces are evidence inputs to it (the evidence-curation gate, decided 2026-07-07 — see *Sonia: The Teacher Agent*). **Attendance is a stamped session attribute (decided 2026-07-07):** the kernel stamps every session — lessons and activity alike (scope stated 2026-07-07, so the attendance-keyed enforcement is anchored on both planes) — **{interactive | scheduled}** from its physical launch path: the User's surface vs the scheduler or the A2A endpoint; for lessons, one name, two attendance states (a second session name would violate one-name-per-write-regime). Invariant, drilled: user-channel events can appear only in interactive sessions — a user-channel event in a scheduled session is an integrity incident. This gives the attendance-keyed enforcement in *Resources as Security* its session-level anchor.

**Composition closure (decided 2026-07-04)**: permissions are closed under composition, not merely per-operation. "Spawn an agent holding Body-write ability" is itself a gated capability. Because all persistent writes funnel through the Applier, a workflow assembled from individually-allowed primitives cannot reconstitute an ungated refinement loop; kernel-resident orchestration quotas bound what an approved workflow can spend at runtime.

**In-flight semantics (decided 2026-07-04)**: a running workflow instance snapshots its definition at launch; approved edits take effect on next launch. The only emergency override is the kernel kill-switch — terminate, never hot-edit.

**Open decision — CLOSED (2026-07-07): accepted-risk while single-user.** The question: a second-opinion channel for high-risk deliberations (connector/scope/workflow class), since Sonia is a single proposer whose context window is the shared failure mode of every component change. Options that were on record: fresh-context re-derivation (same base model), heterogeneous-model review (new egress surface), or accepted-risk while single-user. Decided for accepted-risk: the evidence-curation gate already places a human selection in front of Sonia's entire session-content input domain (kernel signal rows, her own lessons log, and the User's raw channel are the enumerated exemptions) — **curated ≠ vetted** (the User curates relevance, not truth; the couriered consumption caps and the kernel-generated packet fields carry the vetting load) — and the raw channel plus whole-packet hashing stand. Revisit trigger: first hosted/multi-user deployment, the first connector/scope/workflow-class deliberation incident, or any high-risk deliberation premised on couriered content; on trigger, heterogeneous-model review is the preferred option (fresh-context re-derivation second). The original rule stands: this was not allowed to default silently — it was decided.

### Body Persistence & Versioning

The evolved Body needs a home none of the original three components provides.

**Physical home (decided 2026-07-04): one git repository per Kairos instance**, mounted into the sandbox at boot. (The kernel is identical across Kairos instances; the Body is what differentiates this one.)

- Applier's apply = commit; the deliberation-ID — or (2026-07-08) the user-direct edit-event reference — lives in the commit message — audit record and foreign key in one mechanism (commit-signing is deferred hardening; see *Body write-path hardening*). When a deliberation also touches vault grants or memory, the git commit is the **single commit point** of a multi-store apply (see *Cross-store consistency* below); landing the git leg alone is not "applied"
- revert = checkout of an earlier approved commit **+ reconcile the other stores to that commit's lockfile** (never checkout-only; see *Cross-store consistency*) — user-decided, never automatic (see Edit Acceptance Protocol)
- fork = branch (evolution experiments, variance control); merge = a deliberation proposal like any other — no silent branch unions
- Non-file state (OAuth grants, plugin install state) cannot live in git: versioned vault-side, cross-referenced from the repo's lockfile — a "Body version" **is the commit hash** (simplified 2026-07-07: the lockfile is a committed file, already pinned by the hash; it bridges the committed declaration to the vault-side grant versions, and drift between declared and effective state is what boot reconcile detects)
- Memory lives outside the repo — the store of record is **Mem0** (revised 2026-07-09; a vector store, not filesystem-mounted files), reconstructable from the git-committed memory journal (see *Cross-store consistency → Memory leg*); the repo/lockfile carries the memory config — which Mem0 stores exist, their scope labels, and the pinned embedding-model id + index-build hash (not filesystem mount points). The activity plane is read-only against memory via a retrieval-only Mem0 client (no write path), the structural successor to the retired per-store `read_only` mount flag (2026-07-09; see *Memory Design → Decision for SoniaKairos → Agent working memory*) (see the Decouple list)

**Cross-store consistency — apply & revert (decided 2026-07-06, resolves review §2.4/§2.9).** A Body version spans three physically separate stores — git repo, vault grants, memory — and the vault leg is genuinely non-transactional (an OAuth grant lives at an external provider). So the fix is **not** a distributed transaction; it is **one source of truth + idempotent reconciliation** (the GitOps/declarative pattern) — no two-phase commit, coordinator, or compensation log.

- **The git commit is the single commit point and source of truth.** It *declares* the complete desired state: Body structure/components + memory version + the lockfile pinning vault grants and plugin versions. Nothing is live until the commit lands; git commits are atomic, so the version flip is atomic.
- **Apply is commit-last.** Prepare first (provision grants, install plugins, stage memory — all tagged to the pending deliberation-ID, none live) → the commit lands, referencing the prepared state via the lockfile → then cleanup (archive now-unreferenced old grants; hard-delete only for credential revocation). A crash *before* the commit leaves tip unchanged (old version fully live) and any prepared-but-uncommitted vault state orphaned — no lockfile references it, so it is garbage-collected, not left as the untracked live credential of the §2.4 failure. No chimera.
- **Boot reconciles to tip (self-healing).** Every session, before going live, the kernel reconciles actual vault/memory state toward the tip commit's declaration. Satisfiable → live; not satisfiable (a pinned grant missing) → the session is **flagged/quarantined, not run as a chimera**. This is what makes `always tip` safe and absorbs any partial apply.
- **Revert = checkout + the same reconcile** (not checkout-only — that was the §2.9 chimera), two-leg by construction. A **restorability pre-flight** checks every grant the target's lockfile pins is still available; if one was hard-deleted (revoked), revert is *restore-with-declared-degradation* (connectors using it are dead until re-authorized), surfaced to the user — never a silent chimera. So a packet's rollback point carries a live restorability status, and hard-deleting a grant flags every commit pinning it as a degraded rollback target.
- **Memory leg (revised 2026-07-09, Mem0 substrate; supersedes the pivot's collapse-toward-two simplification):** the store of record is **Mem0**, a genuinely separate store again — but its complete write history *is* the git-committed memory journal (each approved memory write is a journal row in the same commit stream), so memory reconciles **from the journal**: boot compares a content digest (op-id set + content hashes, not just a head marker, so tamper is caught) against the journal, and on mismatch or rollback rebuilds Mem0 **from empty**, replaying the ordered journal ≤ target (per op type — add / supersede / delete; see Backend §5 for the replay semantics). Unlike the vault leg (whose external OAuth state can be permanently lost → restore-with-declared-degradation), memory *content* is **always fully reconstructable** (retrieval ranking only to CI tolerance — *Substrate*), so it is the strongest reconcile leg on content. **Memory is the deliberate exception to *Boot reconciles to tip → quarantine*:** where an unreconcilable vault leg quarantines the session, an unreconcilable memory leg *self-heals* by rebuilding from the journal (permitted precisely because it is reconstructable), logged as a flagged drift/tamper event, never a silent self-heal. Three stores again (git · vault · Mem0); only the vault is non-reconstructable. The pivot's premise — that memory being Applier-written and deliberation-gated lets it ride the git commit — still holds; it now rides as the journal row rather than as file content, and the serving copy materializes in Mem0.

**Boot & recovery version policy (decided 2026-07-04): always tip.** Every session — new or resumed via `wake()` — mounts the latest approved commit **and reconciles the other stores to its lockfile before going live** (see *Cross-store consistency*). One enumerated exception: kernel safe mode, trade-off (2) below. Accepted trade-offs, recorded: (1) a resumed session may straddle two Body versions mid-work, so every trace/turn is stamped with the full attribution tuple (body-version × model-id × kernel-version — see *Harness = Kernel + Body*, fixed point 1) and attribution survives; (2) a bad tip affects all sessions until reverted — the machine's kill-switch stops runaway damage, but the revert itself waits for the user (Edit Acceptance Protocol), so a bad tip's blast radius is bounded in cost yet not in duration. One floor under that duration (added 2026-07-06, resolves review §2.10; this is the enumerated exception to *always tip* above): if N consecutive fresh sessions on the same tip die to the watchdog, the kernel drops into **safe mode** — new sessions boot on the kernel floor (kernel stub, no Body mounted), loudly flagged as degraded, and exist to report status and take the user's revert decision, not to continue normal work — instead of an unbounded spawn-and-kill loop while the user is away. This stays on the machine's side of the authority boundary because *withholding* the Body is not *choosing* one: the machine never substitutes an alternative Body version (that would be a machine revert), and the bare floor is the one state that contains no evolved content at all, so nothing unapproved can ride in on it — fixed point 4 already guarantees it is viable. The tip is never auto-reverted; the kernel waits for the user.

**Principal roster (clarified 2026-07-04)**: four principals, two of them human. (Recorded 2026-07-06: at the current stage the two human principals are one person wearing two hats; the channels stay distinct — see *Trust Roots & Principal Authentication*.) A fifth party — the **A2A counterparty** (renamed 2026-07-07 from "served interlocutor") — sits *outside* this trusted four; it is named below and in *The External Channel* (added 2026-07-06, resolves review §4.2; rewritten 2026-07-07), never counted among the principals.

- **Operator** (后台管理者): owns Kairos's kernel and the whole of Sonia (her kernel and her Body) — human-only software process, invisible to end users
- **User**: preference authority — approves/rejects Kairos evolution deliberations; issues instructions to Sonia
- **Sonia** (teacher / Refiner): fixed teacher agent with her own kernel and her own Body, both operator-only; proposes deliberations over Kairos's whole being; runs in **lessons** sessions — see *Sonia: The Teacher Agent* below
- **Kairos** (worker): acts in **activity sessions**; **does not self-modify** (session-local adaptation deferred) and **does not propose** — it produces work and evidence; Sonia is the sole proposer (pivot 2026-07-06)
- **A2A counterparty** (被服务方 — renamed 2026-07-07 from "served interlocutor"; *not* one of the trusted four; added 2026-07-06): a peer agent — or a human behind one — calling Kairos's A2A endpoint when pointed outward (support, intake, agent-internet service). Untrusted, zero-authority — never approves, proposes, or writes memory; its input is evidence at most, stamped `a2a-party`, lower-trust than tool-result. **No instances today** (the endpoint is not yet live); named so its material is never absorbed into the User or tool-result classes the day it exists. See *The External Channel*.

### Sonia: The Teacher Agent (designed after Kairos)

Everything above specifies Kairos — the agent that gets modified. Sonia sits on the other side of the table, and her design is deliberately sequenced after Kairos's: this document pins only her boundary properties.

- **Her own kernel.** Sonia does not run on Kairos's kernel: she has a separate kernel purpose-built for teaching (lesson orchestration — including the scheduling of lessons and dreaming — trace reading, proposal generation and triage). Proposal generation means the proposal's *content*: the argument and the candidate diff; the packet's *rendering* and its machine-generated evidence fields are Kairos-side kernel machinery (see *Landing discipline*). Her kernel's design is a later work item, and Kairos's kernel is not modified to accommodate her — one boundary property pinned now (decided 2026-07-06): her kernel's session-log intake must satisfy the same contract as Kairos's — principal-origin stamping and append-time chaining (see *Trust Roots & Principal Authentication*) — so directives arriving in lessons sessions are stamped the same way; the Applier, the deliberation/audit log, and the gate machinery in front of them (the packet renderer and the trial-run evidence generators) stay on Kairos's side; the teaching pipeline crosses between the two agents only through this gate complex (write direction — the read direction crosses only through the curation-gate filter and the renderer's lessons-log read leg, added 2026-07-07).
- **Second boundary property — the evidence-curation gate (decided 2026-07-07, lessons-input-modes review).** Sonia's kernel reads Kairos's session log only through a kernel-side filter bound to the User's **feed-list** (a user-direct record; kernel enumeration). The filter covers `getEvents()`, trace views, and the trace store; feed granularity is the **whole session** — replay stubs need the complete recorded prefix — with an optional event range as a focus hint. Enumerated exemptions: the User's own raw channel, the kernel's own machinery (watchdog attribution, release canaries, checkpointing), kernel-authored **signal rows** (incident reports, version events, telemetry aggregates — the signal/content separation, see *Refinement Triggers*), and Sonia's own lessons log. The packet renderer double-enforces with the render-time corpus check (kernel enumeration). Selection grants **visibility, never authority**: stamps survive verbatim — a fed session's tool-result is still tool-result, its `a2a-party` still `a2a-party`. Feed-list entries are typed — kairos-trace / external-reference / deliverable (an external-home deliverable is snapshotted into the log at feed time: home × path × content hash) — with **multimodal (image/pixel) feeds deferred**: day-one feeds are text-only, until image events have their own stamp/hash/non-citability rules (deferred 2026-07-07, that definition being the trigger). Accepted cost, recorded with the First Founding Principle's deferral: curation adds latency to offline refinement, merged with the offline-too-slow revisit trigger.
- **Third boundary property — zero egress, ingestion-at-the-gate (decided 2026-07-07, lessons-input-modes review; calibrated against Codex Cloud's default-deny agent phase).** Sonia's plane holds **no egress**. External material enters only when the User feeds a link or file: a **kernel service fetches it once** (GET-only, no credential attached, private/metadata-IP blocking, on the egress-proxy code path), freezing a **hash-pinned artifact** (URL × timestamp × hash × raw bytes) into the curated corpus — the artifact is the raw-channel location and the replayable recording in one. Snapshots render into Sonia's context inside **data-delimited blocks with visible provenance and length caps, never template-evaluated** (presentation discipline; prior art: `docs/research/2026-07-02-sandbox-design-kairos-sonia.zh.md` §7). The rejected alternative, recorded with its adversary: extending the egress registry to Sonia's plane would put live egress inside the most privileged proposer, break evidence fidelity (a page can change between reads), and permit self-curation against Q1(b). Recorded residual: no content-layer sanitization exists (Codex, DeepAgents, and Hermes agree by omission) — injected reference material can still bias Sonia's *argument*; the couriered consumption caps, the kernel-generated packet fields, and user approval are the bounds, and the packet hash pins bytes, not comprehension. A bounded write-time threat scan (detect-and-flag at ingestion and at the Applier's memory-landing boundary; donor located: kairos `MemoryThreatScanner`) was evaluated and **deferred on re-review (2026-07-08)** — the residual stands as recorded. Her user-channel surfaces obey the generalized admission rule (*Trust Roots*).
- **Fixed, operator-only.** Sonia is a fixed teacher agent: her kernel and her Body (versioned by the same persistence machinery as Kairos's) are operator-written and operator-only. She does not evolve in-system, no user channel writes her, and she cannot propose changes to herself — the reviewer must not author her own review function. User complaints targeting Sonia are legitimate input, routed to the operator channel as requirements — which is why instruction extraction classifies every directive's target (Kairos-Body vs Sonia vs kernel; the latter two route to the operator channel) instead of letting Sonia-targeted complaints misroute into Kairos edits.
- **The teaching pipeline.** Persistent modification of Kairos flows through one writer under two triggers (amended 2026-07-08): the teaching pipeline, run jointly by Sonia and the user — **Sonia proposes** (deliberation packet, from lessons sessions) → **the user approves** → **the Applier lands** — or the User's direct edit through the same Applier (*The Applier*, second intake). Sonia is the sole *agent* proposer (pivot 2026-07-06); Kairos produces the activity traces that are her evidence but files nothing itself; nothing lands without a user act — an approval, or the user's own edit.
- **Sessions.** Sonia works in **lessons** sessions; Kairos's **User-curated** activity sessions are evidence inputs to them (qualifier added 2026-07-07; see session taxonomy above).

### Edit Acceptance Protocol

Under open-ended evaluation there is no benchmark to score an edit against. What remains splits into two strata with different fates:

**Objective stratum — preference-independent facts.** Loops, raised exceptions, schema mismatches, stalls, and cost curves are true or false regardless of anyone's taste (the paper's Power Plant incident: one malformed call repeated 842 times over a ~3.5-hour, 1,003-turn loop — unseen by the agent's own reasoning until it finally self-recovered at the very end; detection cannot wait on the agent). The machine owns detection here.

**Preference stratum — everything else.** Whether an edit made Kairos *better* is the user's judgment alone. No score, no judge model, no telemetry ever ACCEPTs an edit.

**Machine authority boundary (decided 2026-07-04): detect and kill, never revert.** On an objective failure signature — **or an unattended session/subtree breaching its spend ceiling (added 2026-07-06, resolves review §4.3; an all-success cost runaway fires no failure signature at all — see *Resources as Security*, where foreground breaches pause-and-prompt and the cross-session daily ceiling halts admission rather than killing in-flight work)** — the kernel watchdog kills the offending session or workflow tree (kill-switch, execution-graph-level — a bad workflow loop spawns N *locally healthy* sub-agents, so single-stream monitoring is not enough) and files an incident report with edit attribution: traces are stamped with the attribution tuple (body-version × model-id × kernel-version), so "which recent edits — or which release or model swap — are prime suspects" is computable, and a swap's behavior change is never misattributed to innocent Body commits (see review §2.5). Body version changes — including revert — are always user decisions; the machine may *withhold* the Body entirely (kernel safe mode — see *Boot & recovery*) but never substitutes one Body version for another. The machine stops the bleeding; the user decides the treatment.

**No probation (decided 2026-07-04).** An approved edit is immediately part of tip; there is no trial state and no exposure-threshold machinery. Rollback is a user-decided checkout of an earlier Body commit — a landed deliberation or (2026-07-08) a user-direct edit. Accepted consequence, recorded: no automatically-maintained "verified" tier exists — the revert target is whatever the user last approved or directly landed, which is acceptable while approvals are personal in a single-user system.

**Pre-apply surface (cheap, kept).** Rejecting a *proposal* is not vetoing an approved state, so the machine may freely gate upstream: schema check and smoke test on every proposed component — necessary but not sufficient (corrected 2026-07-06, resolves review §3.1): the paper's Power Plant failure was **not** a definition-time schema error. The component's definition was valid; each *call* carried a malformed payload — `buttons_to_press: ["Down"]` where the harness requires the literal `["tool"]`, its press-the-tool-button sentinel; the tool itself was named in the separate `tools_to_call` array, which the executor silently ignored with the sentinel absent (Appendix B.3). A proposal-time static check inspects definitions and a smoke test uses well-formed calls — neither ever sees a future malformed payload. The defense that does is the kernel's **runtime action & tool-call schema validator** (see the kernel enumeration), checking both the harness-level action shape and each dispatched call's payload — in the incident the tool was never even dispatched, so a per-tool check alone would still have missed it: malformed actions fail loud, never silently dropped — the only layer that also covers execution no proposal gate reviews. Alongside these gates: fork-based trial runs that replay past sessions (`getEvents()`) to generate before/after behavior diffs as deliberation evidence. Each approved edit carries a machine-readable prediction ("this should reduce failure class X"); later match/mismatch is the only per-edit-attributable calibration signal for a Refiner who never executes the work herself (secondhand refinement). Prediction reconciliation is kernel-owned (decided 2026-07-07): match/mismatch against later failure-class rates is computed by the kernel from its own signal rows and lands as observability rows — the Refiner consumes her calibration signal; she never grades her own predictions.

**Trial-run execution semantics (decided 2026-07-06, resolves review §2.2).** Replay is a kernel mechanism with side-effect isolation — not a re-run. A fork trial has **no live egress**: connector calls are answered from the *recorded* results already in the session event log (record/replay stubs), never re-executed against the world — generating review evidence must not repeat the reviewed session's external effects (replaying last week's session must not resend its emails). Inside the fork, memory is a read-only retrieval view of Mem0 (the retrieval-only client; revised 2026-07-09) and any genuinely filesystem-backed store is copy-on-write: a trial's writes are part of the evidence, never part of production state. And the packet declares its replay fidelity — which calls were answered from recordings, and the point (if any) where the proposed Body diverged from the recorded path so that a stub had no recorded answer. Divergence past that point is marked, so the behavior diff can separate "the edit changed behavior" from "the world moved since the recording", and the evidence states how far it can be trusted. Replay bases are drawn from the User's feed-list only (bound 2026-07-07 — see *Refinement Triggers*; release canaries are operator machinery, exempt), and the fidelity declaration also states **coverage** — how much of the fed corpus bears on this edit, including the honest value "no applicable recorded coverage", in which case the packet's machine-readable prediction stands as the primary falsifiable evidence.

**Open decision — CLOSED (2026-07-07): not built** — whether a rubric-based advisory evaluator (a demoted Outcomes — alarms only, never approves, rubric derived from user preference) exists at all. See *Outcomes* for rationale and revisit trigger; "do not implement acceptance against it" remains binding either way.

### Evolution Deliberation Channel & Preference Charter

The Sonia-proposes architecture makes proposal→review the governance shape of *all* evolution, and open-ended evaluation puts *all* acceptance authority on this same channel. It is therefore the heaviest-loaded component in the system and must be designed as one.

**Deliberation packet — standard contents.** Every proposal ships with:

- before/after **behavior diff** from fork trial runs over replayed sessions — the reviewable form of a change is behavioral, not a 30-file text diff — together with its kernel-generated **replay-fidelity declaration**: which calls were answered from recordings, and where replay diverged from the recorded path (see *Trial-run execution semantics*)
- **per-refinement cost** — under open-ended evaluation, cost is the only objectively measurable scalar in the system; it appears on every packet. That same measurability makes it the adversary's weapon: a *reported* scalar is not a *governed* one, so the enforced spend ceiling that thresholds it lives in *Resources as Security* (added 2026-07-06, resolves review §4.3)
- **rollback point** (the commit to return to)
- a **non-behavior delta declaration, generated by kernel code from the diff itself** (revised 2026-07-06, resolves review §2.10 — never authored by the proposer: a self-reported danger declaration would quietly invert "declared, not inferred" into self-report): scope/credential/egress/resource changes, **plus codification direction** — any logic moving from prompt down into workflow/skill (see *Codification gradient*) — **plus the scope-mismatch flag**: any edit whose landed scope is wider than its cited evidence's scope (added 2026-07-06, resolves review §4.2; see *The External Channel*). The most dangerous properties of connector/workflow edits show *zero* behavioral difference on replayed samples (the ends-vs-means gap), and a teaching user cannot review a workflow diff — so the means are declared by machine and checked against the kernel's static policy gates
- for directive-origin proposals: the **verbatim quote** of the user instruction plus its session/turn reference — without it, the only person who can verify the extraction has nothing to compare against, and a misheard directive enters with the user's own authority stamped on it. The quoted event must additionally carry the kernel's **user-channel origin stamp**, verified by the packet renderer at render time (decided 2026-07-06): the quote proves the text is in the log; the stamp proves it entered through the user's channel — not that the user authored it (the pasted-attacker-text residual is recorded in *Trust Roots & Principal Authentication*). Extended 2026-07-07: the reference is resolved through the global event namespace (plane × session × event) via the renderer's lessons-log read leg, and the quoted event must carry the **authored** sub-stamp — couriered text is refused as a directive source

**Atomicity & partial approval.** A packet declares its internal dependency graph and atomic groups. Partial rejection does not land a remainder: the Applier only lands whole atomic groups, and a partially-rejected proposal goes back to Sonia to re-derive a self-consistent sub-proposal. (Approving a rewritten dispatcher while rejecting the deletion of two sub-checks would otherwise produce a chimera Body nobody proposed or tested.)

**Raw channel (decided 2026-07-04).** The user has direct read access to raw traces and sessions — the lessons log and ingested artifacts included (extended 2026-07-07) — and every piece of evidence in a deliberation packet must link back to its raw location. Without an unmediated channel, "Kairos regressed" and "Sonia's narration drifted" are indistinguishable — the teacher would be the only narrator of her own teaching.

**Preference charter (decided 2026-07-04): user-authored first version.** A declarative, versioned statement of the user's taste, written by the user from intent — not distilled by Sonia from observation. It lives kernel-side (user-owned; neither agent writes it). Sonia's triage — which candidates become proposals, in what order — must cite charter clauses; Sonia may *propose* amendments from observed corrections, but every revision is made by the user and is itself a visible event. Rationale: Sonia's triage is mathematically a reward model whether or not one is declared; the charter is the difference between a versioned, auditable one and a shadow one living silently in her context.

**Triage is decision-shaped, so triage leaves a ledger (decided 2026-07-06, resolves review §2.6).** With no approval tiering, the binding selection happens *before* the user sees anything: what Sonia never surfaces, the user never governs — the drop-set, not the queue, is the de-facto evolution strategy. So every candidate Sonia considers gets an append-only **disposition entry** — proposed / deferred / dropped, with the charter clause cited — appended through the kernel's ledger intake to the kernel-side audit log (Sonia can cause an append, never an edit; see the observability/ledger row class under *Immutable Kernel*), and the user can sample the drop-set as easily as the queue. Citations are spot-checked against the raw channel (are the cited clauses actually apt?), and the charter gets a standing review cadence of its own, decoupled from correction events, so it cannot go stale silently while triage keeps citing it.

**Approval tiering (decided 2026-07-04): none — every proposal is user-approved.** No delegation, no auto-approval class. Accepted costs, recorded: the queue accumulates (the paper shows refinement demand never converges); late approvals go stale (evidence traces and the target Body drift under them); and gradeability skew — cheap cosmetic edits get consumed while structural rewrites starve for attention. Revisit trigger: when the queue visibly starves structural work. The trigger's metric definition is **deferred on re-review (2026-07-08)** — a dual-label scheme (kernel-derived + Sonia-triage; prototype-round lead C.1) was evaluated, not adopted; the trigger stays qualitative.

**Instruction extraction gate.** Every directive extracted from conversation is classified twice before it can justify a proposal: **scope** (one-off complaint / this-session instruction / standing policy — over-generalization turns a passing remark into a permanent system-prompt edit) and **target** (Kairos-Body / Sonia / kernel — Sonia-targeted and kernel-targeted feedback route to the operator channel as requirements; see Principal roster and *Harness = Kernel + Body*). The gate has an executor and a paper trail (decided 2026-07-06, resolves review §2.10): the classification *judgment* is Sonia's — she is the rostered principal whose job reading directives already is (the same test that keeps dreaming rostered; see *Dreaming*) — but the *routing* is kernel mechanism: both labels are appended through the kernel's ledger intake to the audit log, and any directive labeled Sonia-targeted **or mentioning Sonia at all** is auto-forwarded to the operator channel by the kernel, raw text attached, regardless of her label. The reviewer may classify complaints about herself; she cannot route them, suppress them, or summarize them on the way — and a mislabel stays auditable after the fact. Upstream of both labels (decided 2026-07-06): a directive can only be *cited* from a user-channel-stamped event — enforced by the packet renderer, not by restricting what Sonia reads beyond the curation gate (within the curated corpus — parenthetical updated 2026-07-07 — tool-result text, and, outward-facing, a2a-party text, stays available as evidence; it just cannot be quoted as the user's instruction — see *Trust Roots & Principal Authentication* and *The External Channel*). A standing-policy extraction that would land **agent-global** is additionally subject to the scope-mismatch check (see *The External Channel*): an over-generalized directive cannot skip the gate that governs every other agent-global write.

**Preference-premise provenance.** Each accepted edit archives which user instructions/corrections justified it. When a new instruction contradicts an old one, the reverse index surfaces every standing edit premised on the old preference for re-review — preferences have no external invalidation feed; the instruction stream itself is the only signal that a premise died.

**Open decision (deferred)**: counterfactual blind comparison. Under preference-only evaluation the user only ever compares "now vs remembered yesterday", never "now vs never-evolved" — net-negative evolution is structurally invisible. The only known probe is a periodic blind pairwise judgment of {evolved, frozen} outputs on the same task; whether to run it, and how often, is deferred.

### Component Lifecycle & Telemetry

The paper's number to remember: **~95% of the skills the agent authored never succeeded once** (Figure 16's create-and-forget funnel, aggregated across both games and phases: 648 created → 27 ever succeeded — e.g. Emerald from-scratch: 99 → 5; Red bootstrap-continued: 335 → 14; the ~95% is arithmetic over the funnel, not a figure the paper states itself). A self-extending library without lifecycle machinery is mostly noise.

**Per-component telemetry.** Every component in all six classes — third-party plugins included — carries objective counters: invocations, exceptions, latency, cost. "Success" is defined at the execution level only (completed without error); task-level quality belongs to the user's preference channel and never enters telemetry. Without counters, the dead tail is not merely untriaged — it is invisible. Counters have a writer and a home (decided 2026-07-06, resolves review §2.7): the **kernel derives them from events it observes itself** — tool-call dispatch, sandbox exit codes cross-checked against watchdog failure signatures — into kernel-written, agent-unwritable data rows; a component's self-reported exit status is advisory, never ground truth ("completed without error" is the kernel's observation, not the component's claim about itself — a measurement plane the measured code can write is not a measurement plane). This minimal counter-event schema — together with the other non-deferred pieces enumerated under *Traces* (attribution tuple, principal-origin stamp, integrity chain) — is deliberately the *non-deferred kernel* of the deferred trace design, which reconciles the v2.5 prerequisite with the Traces deferral: counters ride the session event log as kernel-stamped events; everything richer stays deferred. Aggregated counters are kernel-origin **signal rows** and reach Sonia's view through the signal/content separation (2026-07-07; see *Refinement Triggers*) — lifecycle triage does not require feeding the underlying sessions.

**Deletion semantics (decided 2026-07-04): archive, never destroy.** The convergence principle extends to executable components as a third state: archived components leave retrieval and loading but remain in audit history, recoverable. This avoids both failure modes at once — unbounded accumulation polluting retrieval, and the paper's incident of deleting a working tool "to make space". Exemption: credentials and OAuth grants are hard-deletable; revocation must actually revoke — revocability outranks history.

**Displacement forgetting.** New components silently shadow proven ones — with no delete event to audit. The paper's evidence, scoped to what it measured (corrected 2026-07-06, resolves review §3.3): in Red bootstrap-continued runs, inherited **sub-agents** fell to a 6.4% (±5.7, n=3) usage share — while inherited skills held 96.5%, memories 100%, and every Emerald store stayed ≥98% (Table 2; §C.2 calls sub-agents "the only anomaly") — and those runs' performance dropped below the no-evolution baseline. One store type, one game, seed variance on the order of the mean: displacement is a demonstrated *risk*, not a demonstrated norm. The counter-rule stands on that basis: authoring a new component requires a reuse check against the existing library (task-signature dedup) inside the proposal — the dedup listing is a kernel-generated packet field (similarity over the existing library; decided 2026-07-07), never the proposer's self-report.

**Edit semantics per component class.** Each class declares its mutation grammar: what may be CRUD-edited, what is wholesale-replaced, what is append-only. A structural rewrite is its own reviewed operation class, not merely a large edit. (Connector rows: credentials never evolve with the component. Plugin rows: pinned; upgrades arrive as proposals.)

**Codification gradient.** The same behavior can live in the prompt (model-mediated at every call — bad content still passes through the model's judgment) or in a workflow (deterministic — the model never sees the decision again). Moving logic down the gradient removes the model's in-loop error correction, and dreaming-style consolidation naturally pushes that way, because every step looks like an efficiency win. Rules, each with a mechanism and an owner (revised 2026-07-06, resolves review §1.6): codification requires demonstrated stable repetition — and because a prompt→workflow move can show *zero* behavior difference on replay (same outputs; what changed is that the model's in-loop judgment left the path), the direction of movement is **flagged by the kernel-generated delta declaration** (static analysis marks any logic descending from prompt into workflow/skill; see the packet contents), not left to the proposal's self-description — trial-run evidence alone is structurally blind to it. The **never-codifiable decision classes live as a kernel-hosted governed data row** (agent-unwritable; amended only via deliberation → user approval → Applier), so the list has a home and a change gate instead of being prose. And a base-model upgrade triggers a **de-codification review**, owned by the operator as a mandatory item on the model-swap checklist (see *Operations*) — codified logic is a control-flow patch for the *old* model's weaknesses.

### Refinement Triggers, Cadence & Task Sources

**Triggers run in parallel, not on a clock alone:**

- **objective failure signatures** from live traces (loops, repeated tool failures, stalls) — attacker-influenceable, so each carries the principal-origin attribution of what induced it and its kill stays bounded to the offending subtree (the anti-induction margin; added 2026-07-06, resolves review §4.3 — see *Resources as Security*) — the evidence that overnight-only refinement can be too late is a *single case observation*, cited as such (corrected 2026-07-06, resolves review §3.4): one GPP run (Pokémon Yellow Legacy, a human-in-the-loop predecessor system, paper Appendix B.3) where self-repair stopped after a ~500-turn stall — not a cross-run measurement. The paper's added mid-run mechanism is a scheduled *out-of-band* Refiner pass (every F steps) that does not depend on the acting agent self-editing — the paper's agents do keep live meta-tool edit access alongside it — and the out-of-band pattern is what this trigger implements: since session-local adaptation is **deferred** (see *First Founding Principle*), a live failure signature triggers an out-of-band proposal (and the kill-switch on objective failure), never an in-session self-edit. **Signal/content separation (decided 2026-07-07, lessons-input-modes review):** kernel-origin **signal rows** — incident reports, version events, telemetry aggregates — reach Sonia's view directly, a named exemption from the evidence-curation gate (kernel-origin is already an admissible fact source); the **offending session's content** enters only through the User's feed-list, never auto-fed — signatures are attacker-influenceable, and auto-feeding would hand the curation key to whoever can induce a signature; the User is paged (the *Resources as Security* precedent) with a one-click feed. A signal alone can justify proposing *that evidence is needed*; an edit citing session content requires that content fed
- **user real-time instructions** — the natural hot path; with all-personal approval, expedited deliberation is just a proposal the user approves in the moment, no extra machinery. These arrive on Sonia's lessons plane, in an **{interactive}**-stamped session, through the user-channel surface (arrival plane stated 2026-07-07; see the session taxonomy)
- **scheduled dreaming** — the offline consolidator; its outputs are deliberation proposals like everything else

**Task sources for trial runs (open decision, deferred 2026-07-04).** For now, fork trial runs replay recent **feed-listed** sessions (bound 2026-07-07 to the evidence-curation gate). The alternative — a rolling probe set curated from historical user corrections (explicitly a shifting sample, not a benchmark) — is deferred; constraint added 2026-07-07: if ever adopted, the probe set must be **User-curated or user-approved** — a Sonia-curated probe set would contradict the evidence-curation gate. Release canaries are operator machinery, exempt from the gate. Recorded blind spot: a failure the user corrected long ago can silently recur, and recent-session replay will not catch it.

## Operations: Running a Live System (added 2026-07-06)

Everything above governs *change*; a live system also has to survive *time* — storage loss, releases, upstream drift, and its own guard machinery silently rusting. Five decided rules and one scope declaration (resolves review §2.3, §4.4, §4.5):

**Durability & the recovery unit (resolves §2.3).** "The record remains" is a property only if something enforces it; until this section, storage loss was a failure class the design had no words for. The persistent stores are dense with cross-references — commit ↔ audit deliberation-IDs, lockfile ↔ vault grants, annotation events → prior event IDs, the Body repo's memory config → mounted stores — so no store can be restored alone. The unit of backup and restore is the consistent tuple **(Body commit hash × lockfile × vault grant versions × memory store state × session-log offset × log chain heads)** (chain heads added 2026-07-06 — restore is one of the moments the logs are re-trusted; see *Trust Roots & Principal Authentication*), snapshotted on a schedule (the cadence is the RPO) and restore-tested on a schedule (the rehearsed restore time is the RTO) — an untested backup is a hypothesis, not a backup. A restored tuple passes the same boot reconcile gate as any tip (see *Cross-store consistency*): satisfiable → live; not satisfiable → flagged, never run as a mismatched hybrid. If, while single-user, the user prefers to accept storage-loss risk instead of running backups, that is recorded as an explicit trade-off like every other in this document — what is not acceptable is the silent absence of the decision.

**Guard machinery is exercised, not assumed (resolves §4.4).** The watchdog, kill-switch, and Applier are the components everything else trusts and nothing above tests — and a kill switch that has never been pulled must be presumed broken. Standing rule: scheduled drills — synthetic failure signatures the watchdog must catch, dry-run applies the Applier must land and must *refuse* correctly (extended 2026-07-08 with the user-direct intake: a seeded direct edit whose edit event lacks the user-channel/authored stamp must be refused), a periodic kill-switch pull against a sacrificial session, and (added 2026-07-06) a seeded non-user-channel directive quote the packet renderer must refuse, a corrupted log segment the integrity chain must flag, a forged cross-origin request the approval surface must reject, and (added 2026-07-06, resolves review §4.3) a synthetic all-success spend runaway the spend-watchdog must halt at the ceiling plus an induced failure signature whose bounded-to-subtree kill and origin attribution must both be verified — and the kernel services emit their own liveness/health telemetry into the kernel-written observability rows (see *Immutable Kernel*), watched from outside themselves. The observability vocabulary stops being Body-only: a kernel release that quietly breaks the watchdog's signature matcher should be caught by the next drill, not by the next real incident. Standing coupling rule (added 2026-07-07): admission of any new guard mechanism into the kernel enumeration requires a matching drill added to this roster in the same change — a guard without a drill is presumed broken from day one; first application, the lessons dialogue surface's admission-rule target (*Trust Roots*).

**Kernel releases and model swaps are the largest behavior changes the system experiences (resolves §4.4; gate for review §2.5).** Body_t was evolved against kernel K and model M — months of edits, codified workflows, and memory-navigation habits are fitted to their context assembly and semantics — yet releases arrive from outside every in-system gate. They stay operator acts (that is the kernel's definition), but they get the discipline Body edits already have: every kernel release and model swap **stamps an in-system version event** on the session log — carrying the release's identity and a changelog/source reference (added 2026-07-06), so "a release happened" *and* "what changed" are recorded facts, not folklore (self-attested by the new kernel itself — an honest-operator aid, not a defense; see *Trust Roots → Kernel releases*); traces carry the attribution tuple (body-version × model-id × kernel-version) so incident attribution can tell a bad Body edit from a bad release; a release is **canaried against replayed recent sessions** before fleet-wide rollout (the decided trial task source — adopting a curated probe set instead would pull forward the deferred decision under *Refinement Triggers*; canary replay is operator machinery, exempt from the evidence-curation gate — noted 2026-07-07), since behavior diffing was built for Body changes and this is the change class it is blindest to; and a model swap is an operator-checklist action whose mandatory items include the **de-codification review** (see *Codification gradient*). Attributable and rehearsed — not in-system governed.

**The outside world changes on its own clock (resolves §4.4).** Pins and gates assume change waits for approval; vendor deprecations and urgent upstream fixes do not. The lockfile therefore has an **expedited lane**: an urgent upstream change (a deprecation deadline, a critical fix to a pinned dependency) enters as a pre-templated, queue-prioritized deliberation — still user-approved, never auto-landed — and a pin known to be against an urgently-flagged version carries a visible staleness flag until its deliberation resolves. Pinning exists to stop *ambient* change, not to freeze known-bad state with the machinery working exactly as designed. (Credential rotation and expiry — the time dimension of secrets — is deferred with Tier 2; see *Body write-path hardening*.)

**Machine authority is symmetric across the two agents (resolves §4.4).** "Detect and kill, never revert" was derived at Kairos's table and never re-derived for the second agent. Sonia's lessons plane gets the same objective-stratum floor: her kernel runs its own watchdog, orchestration quotas — including the money axis (spend ceilings per dreaming pass and per replay batch; added 2026-07-06, resolves review §4.3; see *Resources as Security*) — and kill-switch over lessons sessions — an unattended overnight trial-run loop that stalls, thrashes, **or runs the bill past its ceiling** is killed exactly like an activity-session loop, and because a trial-run invocation carries its lesson's session ID, killing the lesson also cancels the Kairos-side fork replays it invoked through the gate complex — and her incidents file to the operator channel, since her Body is operator-owned and the user is not her reviewer.

**Task deliverables are externally homed (scope declaration, resolves §4.5).** The seven components describe the agent; none of them is the *work product*, and an unnamed thing gets no policy. Decided: deliverables (the report, the patch, the running service) are **not** an eighth component — by contract they live in external homes (work repos, user-designated stores), and a task is not "done" until the product is deposited there. The sandbox is never a deliverable's home: it is disposable by design, and "never auto-destroy" protects the agent's definition and memory, not files in a container a TTL will reap. Long-running tasks deposit incrementally, so a mid-task TTL costs minutes of work, not days; the kernel flags any session that dies holding an undeposited declared deliverable. The session log records the *process* forever; the *product's* durability belongs to its external home — declared here as a boundary of this architecture's guarantees, so it is a stated contract rather than a silent hole.

## Open Source Reference: LangChain Deep Agents, NousResearch Hermes Agent

Architecturally, Deep Agents has many parallels with Managed Agents:

- **Loop and sandbox separation**: An agent can run inside the sandbox, as with Claude Agent SDK, or run in a long-lived container and control a remote sandbox over the network. A sandbox failure does not affect the agent loop.
- **Persistent conversation records**: A checkpointer provides short-term memory at the thread level, with PostgreSQL used in production. The loop and record are separate, so the loop is stateless and can scale horizontally.
- **Long-term memory across sessions**: A Store provides cross-conversation persistence with semantic search through pgvector. Namespaces can separate users and projects. Memory is mounted into the agent's virtual file system as files.
- **Credential isolation** (corrected 2026-07-07, credential-isolation study — the earlier "auth proxy" claim was refuted at code level): architectural, not proxy-based. No auth proxy exists in the codebase (the docs list credential-injecting proxies as a provider-side option "not yet widely available"); MCP and model keys stay out of the sandbox only because credential-consuming work runs in the harness process. The documented setup-script happy path, meanwhile, writes the git token into the sandbox's `.git/config` — the same pattern this charter's work-repo bullet records as a residual — on the same page where the docs warn "Never put secrets inside a sandbox."
- **Multi-agent orchestration**: Tasks are delegated to subagents, each with its own context window. Deployed agents can call each other through MCP and A2A (the outbound direction of *The External Channel*).

Deployment is also more flexible: fully managed (LangSmith Cloud), hybrid (control plane in the cloud, data plane on self-managed infrastructure), and fully self-hosted (the whole stack on self-managed Kubernetes) are all supported.

**NousResearch Hermes-agent** (section completed 2026-07-07 — the title promised it; evidence: `docs/research/2026-07-07-credential-isolation-codex-deepagents-hermes.md`): a host-resident, single-tenant harness whose reference value is three postures rather than architecture. First, an unconditional, fail-closed env strip-list at every child-spawn boundary — no skill or config may re-allow a provider credential (adopted as a deferred candidate in *Work-credential hardening*). Second, zero egress as an explicit opt-in air-gap (`--network=none`) rather than a default — the posture this design inverts (*Sandbox egress* is default-deny). Third, unusual honesty that env scrubbing "reduces casual exfiltration. It is not containment" — the does-not-confine discipline this charter's recorded residuals follow. It never writes tokens into git remote config; its redactor treats token-in-URL as a secret pattern to scrub.
